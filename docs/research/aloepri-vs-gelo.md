## Definitions

| Term | Meaning |
|---|---|
| **AloePri** | ByteDance's covariant-obfuscation scheme. The subject of this analysis. |
| **Covariant obfuscation** | Joint transform of input data AND model weights such that `f̃(x̃, θ̃) ≈ τ(f(x, θ))`. |
| **τ** (tau) | The secret token-level permutation (∈ S_n where n = vocab size). |
| **P̂, Q̂** | Key matrix and its (right-)inverse with P̂·Q̂ = I, generated by Algorithm 1. |
| **h** | Expansion size — extra hidden dimensions added by P̂ (default h=128). |
| **β, γ** | Block-permutation window size and frequency-weighting parameter for RoPE shuffling. |
| **α_e, α_h** | Noise coefficients for embedding and lm_head Gaussian noise. |
| **R̂_qk, Ĥ_qk** | 2D rotary and scaling matrices applied to Q/K weights (RoPE-aware). |
| **Û_vo** | Random invertible matrix paired between V and O projections. |
| **TTRSR** | Text Token Recovery Success Ratio — fraction of original tokens an attacker recovers. |
| **VMA / IA / ISA / IMA / NN / TFMA / SDA** | The seven attacks AloePri's reference code implements (Vocabulary-Matching, Invariant, Internal-State, Inversion-Model, Nearest-Neighbor, Token-Frequency-Matching, Substitution-Deciphering). |
| **LMaaS** | Language Model as a Service — AloePri's deployment target. |
| **MLA** | Multi-head Latent Attention (DeepSeek-V3). |
| **RmDP** | Rényi-metric Differential Privacy — AloePri's formal privacy framework. |

## TL;DR

**AloePri's strong security claim assumes the server doesn't have
the original model weights.** When that assumption holds, the
static key matrices `P̂/Q̂`, Gaussian noise `α_e·ε`, and token
permutation `τ` give the headline guarantees. Under our openweight
deployment, that assumption doesn't hold — but AloePri's defense
**degrades to empirical attack-resistance**, not "no security."
The paper does explicitly test the open-weight case via VMA, which
recovers `τ` "based on the relationship between plaintext and
obfuscated model weights"; reported result is TTRSR < 15% on
Qwen2.5-14B, ≤ 5% on DeepSeek-V3.1-Terminus. So **AloePri provides
weak but nonzero defense under public weights**, primarily from the
baked-in Gaussian noise on `W_e` and `W_h`.

The deeper issue for our use case is **structural, not numerical**.
AloePri is fully static — one `τ`, one set of key matrices, one set
of noise vectors, all baked into shipped weights. The same prompt
sent twice produces identical obfuscated traffic. Observations
accumulate across requests. By contrast, GELO+TwinShield resamples
a fresh Haar `A_b` per forward pass; the same prompt twice produces
different masked activations crossing PCIe; each batch is
information-theoretically independent. **GELO is in a categorically
stronger security class** (per-batch fresh randomness vs static
deployment-time obfuscation), regardless of whether either scheme's
specific TTRSR numbers look acceptable in isolation.

Of the ten distinct AloePri techniques surveyed below, **nine are
inapplicable** to our setting — not because they "don't work" in an
absolute sense, but because they protect the wrong axis (weights,
not per-request entropy) for our threat model. **One — the
empirical attack suite at `src/security_qwen/` — is highly portable
and useful**: ~120 KB of PyTorch implementations of VMA, IA, ISA,
IMA, NN, TFMA, SDA that we can run against `InProcessTrustedExecutor
+ WgpuVulkanEngine` as a regression-grade attack benchmark. **This
is the one piece worth porting** (2–3 weeks).

## 1. Mechanism Summary

AloePri operates entirely **offline** — the client (data owner)
rewrites the model `θ → θ̃` once, ships `θ̃` to the server, then
performs lightweight per-prompt token mapping `tok_i → τ(tok_i)` at
inference time. The server runs **unmodified** transformer inference
on the obfuscated weights using vLLM 0.9.1 / SGLang.

```
Client (CPU-only, two Xeon 8457C):
   τ ∼ S_n                                   # secret permutation
   {P̂_i, Q̂_i} ← Alg.1(d, h=128, λ=0.3)       # one set of key matrices
   ε_e ∼ N(0, σ_e² I), ε_h ∼ N(0, σ_h² I)    # weight noise
   W̃_e ← Π (W_e + α_e ε_e) P̂_e              # embed: token-perm + noise + key
   W̃_q, W̃_k, W̃_v, W̃_o ← Alg.2(...)          # attention: key + RoPE rotation + head perm
   W̃_ffn ← key-matrix transform              # FFN: P̂/Q̂ between layers
   Ship θ̃ = {W̃_e, ..., W̃_h} to server       # one-time deployment

Online per prompt:
   tok_ids → [τ(t) for t in tok_ids]         # cheap O(seq_len) map
   server runs vLLM(θ̃, permuted-ids)        # unchanged inference
   response_ids → [τ⁻¹(t) for t in response_ids]
```

Key matrix generation (Algorithm 1, `src/keymat.py:82-146`):

```python
B = U + λV         # U ∼ Haar(O_d), V ∼ N(0, 1/d)^(d×d) — invertible
E = E_1 E_2        # low-rank d×h
F = F_1 F_2        # low-rank h×d (paired with E so E·F-related quantities are bounded)
Z ∼ Haar(O_{d+2h}) # outer orthogonal randomization

P̂ = [B C E] Z      where columns of C come from null(F^T)
Q̂ = Z^T [B⁻¹ F D]^T where rows of D come from null(E)

Identity: P̂ · Q̂ = I_d  (proven by the null-space construction)
```

The d×d hidden state, after right-multiplication by P̂, lives in a
(d+2h)-dim space whose two extra "expansion" axes carry decoy
content that cancels via Q̂ on the next layer.

For attention (Algorithm 2), the construction adds three more pieces
on top of P̂/Q̂:

- **R̂_qk**: a 2D rotary matrix applied to Q and K, commuting through
  RoPE because both are 2×2-block-diagonal.
- **Ĥ_qk**: diagonal scaling on Q and K (s_i and s_i⁻¹ paired
  between Q and K), preserving Q·K^T.
- **Ẑ_block**: block permutation of RoPE pairs within a dynamic
  window of size ≤ β, weighted by RoPE frequency parameter γ.

Plus **inter-head permutation** (τ_kv shuffles K/V heads at the
single-head level; τ_group shuffles Q/O within groups).

For RMSNorm, the diagonal scale `w_norm` is permuted/rescaled with
an approximation error `e_C^norm` from the changed normalization
denominator (now over d+2h dims).

## 2. Threat Model — The Critical Divergence

### 2.1 Side-by-side

| Property | AloePri (§3.2) | GELO+TwinShield (`gelo.md` §2) |
|---|---|---|
| Adversary | Honest-but-curious server | Honest-but-curious GPU + host operator |
| Adversary's location | Cloud datacenter | Same physical host as TEE (PCIe-attached) |
| Adversary's knowledge of model | Only obfuscated `θ̃` | Full public `θ` (openweight) |
| What crosses an untrusted boundary | Token IDs (permuted by τ) | Hidden states (masked by Haar A) |
| Per-request entropy | None — static key | Fresh Haar `A_b` per forward pass |
| Trust root | Trusted client device | Trusted SEV-SNP silicon |
| Privacy framework | Rényi-metric DP (empirical attack-resistance) | Information-theoretic (Haar uniform = perfect mixing) + integrity probe (Freivalds) |
| Online cost | <10% TPOT increase at h=512 | 6–27% wall-clock (depends on path) |
| Offline cost | Rewrite all weights (one-time) | None |
| Engine modification | None (vLLM unchanged) | Required (TEE-in-loop per offload) |

### 2.2 How AloePri degrades under our open-weight assumption

The paper's threat model (§3.2) says: *"The attacker can access the
model and observe the service's inputs and outputs."* This is
ambiguous between obfuscated-only access and full plaintext-weight
access — but the experimental section resolves the ambiguity. **The
VMA experiment explicitly assumes the attacker has both `θ` and
`θ̃`**: it is defined as recovering the mapping "based on the
relationship between plaintext and obfuscated model weights." So
AloePri *does* test the open-weight scenario, and reports:

| Model | VMA TTRSR | IMA TTRSR | ISA TTRSR |
|---|---|---|---|
| Qwen2.5-14B-Instruct | < 15% | < 15% | < 15% |
| DeepSeek-V3.1-Terminus (671B) | ≤ 5% | ≤ 5% | ≤ 5% |

So AloePri **does provide some defense under open weights**, primarily
via the baked-in noise `α_e ε` on `W_e` (and the analogous noise on
`W_h`). The key matrices `P̂_e/Q̂_e` are nearly orthogonal with low-rank
perturbation: at default h=128, λ=0.3, d=8192, `P̂_e · P̂_e^T ≈ 1.016 I`
(see Algorithm 1 — `B = U + λV` is near-orthogonal; `C` is sampled
from `null(F^T)` with C·C^T ≈ (h/d)·I; `E` is low-rank). So per-row
matching (VMA) on `W̃_e` against `W_e` would succeed *trivially* if
the only protection were P̂_e. The defense against open-weight VMA
comes from the noise: at α_e=1.0 (noise std equal to weight std),
matching is reduced to a Gaussian denoising problem where ~85% of
rows are unrecoverable.

**So the literal "AloePri provides no security under open weights"
claim is incorrect.** It provides empirical defense at TTRSR 5–15%
against the specific attacks the paper tests, by accepting:

1. **Some prompt leakage as the design point** — 15% of tokens in a
   100-token medical prompt is enough to reconstruct most content
   words and recover intent.
2. **Static defense** — `τ`, `θ̃`, and the noise are baked once.
   Same prompt twice → identical obfuscated traffic.
3. **No formal guarantee against adaptive attackers** — the paper
   tests the 7 published attacks individually; an attacker that
   composes VMA + TFMA + IMA + accumulated observations across
   many requests is not in the threat model.

### 2.3 Why static obfuscation is the wrong primitive for our threat model

AloePri's degradation under open weights is one issue; the deeper
structural problem for our use case is **the lack of per-request
fresh entropy**, regardless of whether the model is public or
private. The same prompt sent twice through AloePri produces:

- The same obfuscated token-ID sequence (τ is fixed)
- The same obfuscated hidden-state trajectory through θ̃
- The same obfuscated output token sequence

An adversary collecting many obfuscated requests can correlate
across them, learn distributions of "frequent" vs "rare" obfuscated
tokens via TFMA, and compose VMA with that frequency-learned prior.
The paper's individual-attack TTRSR numbers do not bound the
combined-attack rate.

By contrast, GELO+TwinShield resamples `A_b ∼ Haar(O(n+k))` per
forward pass. The activation crossing PCIe in batch `b` is one
specific masked tensor `U_b = A_b · stack(H, shield_rows)`. In
batch `b+1` (same prompt), `U_{b+1}` is a *different* masked tensor
under a freshly drawn `A_{b+1}`. Information-theoretically, the GPU
observing `(U_b, U_{b+1})` cannot distinguish "same prompt twice"
from "two unrelated prompts" — both look like independent random
rotations of *some* hidden states. Multi-batch attacks like ICA on
the Gram matrix `U^T U` are then blocked by the shield rows
(TwinShield's defense; `gelo.md` §3.3).

**This is a categorical difference**: empirical defense (AloePri,
under a specific attack budget) vs information-theoretic mixing per
batch (GELO). Even if AloePri's TTRSR 5–15% were acceptable in
isolation, the static-key family is in the threat-model regime
that the precomputed-basis paper (arXiv 2602.11088) called out as
concerning. GELO avoids that family entirely by construction.

### 2.4 What about non-static randomness?

For completeness: AloePri does NOT propose per-batch or per-prompt
resampling. The paper's online phase is a deterministic O(seq_len)
token-ID map; the model `θ̃` is reused for every request. The
authors note in §8 that the construction could be extended to
"scenarios where only the server can access the model" via CPU-TEE
or FHE, but the *core* mechanism is offline static rewrite with
deterministic online operation. This is the structural property
that pulls AloePri into the same family as STIP, ObfuscaTune,
SOTER, TSQP, and TransLinkGuard — schemes that may or may not have
a known attack but share the static-key property.

## 3. Technique-by-Technique Applicability

Each AloePri primitive evaluated under our four user-specified axes
(performance, scalability, accuracy, threat-model fit) AND our
specific open-weight constraint.

A note on the threat-model column: "doesn't apply to us" below means
the technique protects a leak channel we don't have (e.g., token
IDs crossing PCIe in our design — they don't), or addresses an
attack our existing primitive already handles with a structurally
stronger property (per-batch freshness vs static-deployment-time).

| # | AloePri primitive | Performance | Scalability | Accuracy | Threat-model fit for our setup | Verdict |
|---|---|---|---|---|---|---|
| 1 | Token-level vocab permutation `τ` | Free | Free | Bit-exact | Doesn't apply — token IDs are TEE-internal in our design; embedding lookup never crosses PCIe. | ✗ Skip |
| 2 | Embedding/lm_head Gaussian noise `α_e ε` | Free (offline-baked) | Same | 0–3% loss | Targets server-side `W_e`-recovery (VMA). In our design the GPU has `W_e` directly. Adding noise costs accuracy without changing what our adversary can already do. | ✗ Skip |
| 3 | Key matrices P̂, Q̂ with expansion h | +5% GPU compute at h=128 (5120→5376 dims) | Linear in h | Bit-exact (null-space cancels) | The "extra dims" role is filled by our shield rows with per-batch freshness — strictly stronger. Adding P̂ static expansion on top adds cost without security gain. | ✗ Skip |
| 4 | Block-RoPE permutation `Ẑ_block` | Free | Free | Approximation error (paper-bounded) | Static defense against a `Q_W̃` ↔ `Q_W` matching attack; not the attack class we face (we mask activations, not weights). | ✗ Skip |
| 5 | RoPE-aware Q/K rotation `R̂_qk` | Free | Free | Bit-exact | `R̂ R̂^T = I`, so it cancels in Q·K^T. Doesn't hide the dot product the GPU computes. Useful only if adversary tries to recover Q or K *individually* from weight inspection, which is not our attack surface. | ✗ Skip |
| 6 | Q/K scaling `Ĥ_qk` | Free | Free | Bit-exact | Same as #5; cancels in Q·K^T. | ✗ Skip |
| 7 | V/O paired invertible `Û_vo` | Free | Free | Bit-exact | Same cancellation pattern in V·O composition. | ✗ Skip |
| 8 | Inter-head permutation `τ_kv, τ_group` | Free | Free | Bit-exact (heads are exchangeable) | Permutes static weight storage. Adversary with reference public weights matches heads by content trivially. | ✗ Skip |
| 9 | RMSNorm `κI` approximation | Free | Free | Lossy (`e_C^norm` ≈ small) | Compensates for the d+2h dimension change introduced by P̂ expansion. We're not introducing that expansion, so we don't need the compensation. | ✗ Skip |
| 10 | Empirical attack suite (`src/security_qwen/`: VMA, IA, ISA, IMA, NN, TFMA, SDA) | N/A (eval-only) | N/A | N/A | **Highly useful** — attacks are model-agnostic and apply equally to GELO. The reference code is the most thorough open-source attack suite for split/obfuscated inference. Currently we have no empirical attack benchmark. | ✓ **Port** |

**A note on what "skip" means.** The skip verdicts above are not
"these techniques don't work." Techniques 2 + 3 + 8 in particular
provide *some* defense — they're how AloePri achieves its TTRSR
5–15% under open weights. We skip them because (a) the leak axis
they protect is not in our threat model, (b) our per-batch Haar
mask + shield rows defends the *same* leak axis with information-
theoretic per-batch mixing rather than static empirical resistance,
and (c) stacking AloePri's offline machinery on top of GELO would
cost performance/accuracy without adding to GELO's per-batch
freshness property.

**Aggregate accuracy comparison.** AloePri on DeepSeek-V3.1-Terminus
(671B) reports 0.0–3.5% accuracy loss against plaintext. GELO on
Qwen3-Embedding-0.6B reports bit-exact `top1_vs_plain = 1.00`
(`gelo.md` Appendix). AloePri's loss is the price they pay for static
RoPE block permutation + RMSNorm approximation + Gaussian noise on
embeddings — protections we don't need. **We win on accuracy.**

**Performance.** AloePri's <10% TPOT figure is achieved because the
entire LLM runs on the untrusted side as a vanilla forward pass. Our
6–27% overhead exists because TEE is in-the-loop per offload (mask
sample, mask apply, GPU call, U-Verify probe, mask unapply, shield
strip). The two numbers measure **different protocols** — AloePri can
be that fast precisely because it abandons the per-forward freshness
that gives GELO its security under public weights. The comparison is
apples-to-oranges.

**Scalability.** AloePri scales linearly with vocab size (`τ` storage
+ `W̃_e` rewrite) and h (extra GPU dims) but per-request cost is
constant. GELO scales with hidden_size² per forward (Householder
sample) but is independent of vocab size. For an LLM-generation
workload at 128k context, AloePri's static rewrite scales better;
for an embedding workload at n≈400, GELO's per-batch cost is small
in absolute terms (~9 ms/forward, `gelo.md` §8).

## 4. The One Win — Empirical Attack Suite

`src/security_qwen/` in the AloePri repo contains:

| File | Size | Attack |
|---|---:|---|
| `vma.py` | 30 KB | Vocabulary-Matching — recover `τ` by matching obfuscated embedding rows to known vocab embeddings |
| `ima.py` | 34 KB | Inversion Model Attack — train a small model to invert hidden state → token id |
| `isa.py` | 20 KB | Internal State Attack — extract prompt from layer-i hidden states |
| `tfma.py` | 14 KB | Token Frequency Matching — recover `τ` from corpus frequency statistics |
| `sda.py` | 11 KB | Substitution Deciphering Attack — n-gram-based recovery |
| `ia.py` | 1 KB | Invariant Attack — exploit weight-invariants under obfuscation |
| `gate5_scan.py` | 7 KB | Gate-5 sweep harness |
| `gate6_enhancement.py` | 12 KB | Gate-6 enhancement |
| `artifacts.py`, `metrics.py`, `schema.py`, `summary.py` | 17 KB | Result aggregation + reporting infrastructure |

Plus `tests/test_security_qwen_*.py` (~25 KB) wiring the attacks to
Qwen-format checkpoint files.

**Why these matter for us:**

1. The attacks are **model-agnostic** — they take an intermediate
   tensor (hidden state, attention output, embedding) and try to
   recover the input token sequence. They don't care whether the
   obfuscation came from AloePri, GELO, ObfuscaTune, or anything else.

2. We currently have **no empirical attack benchmark** for our GELO
   prototype. `gelo.md` §8 lever #5 listed "an empirical
   attack-resistance benchmark using `qsxltss/Game-of-Arrows` as the
   attack-side reference" as a future spike. AloePri's suite is
   broader and better-engineered than Game-of-Arrows.

3. The attacks target exactly the leaks we care about:
   - **ISA** validates that hidden states crossing PCIe under GELO
     mask are non-invertible (the core BSS-hardness claim).
   - **IMA** validates against an adaptive inverter (the threat from
     arXiv 2507.16372 "Depth Gives a False Sense of Privacy").
   - **VMA / IA** validate that an attacker with public weights *plus*
     observed obfuscated activations cannot recover the token-level
     permutation (the open-weight attack we said disqualifies
     AloePri — running it against GELO would be our analogous test:
     can the adversary recover the activation mask `A` per batch?
     If GELO is sound, ASR should be near random).
   - **TFMA / SDA** validate against frequency analysis on
     obfuscated traffic.

4. The reference code is PyTorch + standard Qwen checkpoint format —
   plug into our existing `gelo-embedder` Python harness directly,
   or port the algorithms to Rust if we want them in-tree.

### 4.1 Porting plan

**Phase 1 (1 week): Snapshot capture.** Add a feature flag to
`InProcessTrustedExecutor` that, when enabled, snapshots
per-forward-pass activations *immediately after GELO mask-apply*
to a `Vec<Array2<f32>>` keyed by `(layer_idx, op_kind)`. This is
what the attacker would observe on PCIe.

**Phase 2 (1 week): Attack harness.** Wire the snapshots into a
new Python eval harness at `evals/aloepri-attacks/` that imports
`src/security_qwen/{vma,ima,isa,ia,tfma,sda}.py` from a pinned
AloePri commit. Run each attack against snapshots from:
- Plain Qwen3 (control: should recover ~100%)
- Qwen3 + GELO mask, no shield rows (control: should recover ~most)
- Qwen3 + GELO mask + shield rows (default: should recover <5%)

**Phase 3 (1 week): Integration into bench-gate.** Add the attack
suite to the release-gate CI (alongside `BEIR_DOCS=500` per
[[feedback-bench-doc-cap]]). Threshold: TTRSR < 10% for ISA/IMA at
the default mask config.

### 4.2 What this empirically validates

- The `gelo.md` §6 claim that per-batch fresh Haar mask makes
  hidden-state inversion infeasible — IMA and ISA give a concrete
  ASR number rather than the current "no published attack" argument.
- The TwinShield shield-row defense against the Gram leak — running
  with shield rows on vs off gives a measurable delta in ASR.
- The robustness of GELO against the **same attack class** that
  AloePri claims to resist. If GELO at TTRSR ≈ 0% matches AloePri's
  TTRSR ≈ 5% under their own attacks, we have strong evidence the
  protocol is at least as strong against the published attack
  literature.

## 5. Anti-Recommendations

Do NOT port these AloePri techniques into the GELO stack:

### 5.1 Static key matrices on weights

AloePri's `W̃ = P̂ W Q̂` construction adds no security under public
weights and would force every GELO offload to compute on (d+2h)-wide
operands instead of d-wide. This **degrades** our performance with no
threat-model benefit. The closest GELO analog — shield rows — already
adds k=8 extra rows per forward with fresh entropy; AloePri's static
expansion is strictly weaker.

### 5.2 Offline noise injection on weights

The Gaussian noise α_e ε on W_e and W_h targets the weight-recovery
attack. Under public weights, the adversary doesn't need to recover
W_e — they already have it. Adding offline noise just degrades
accuracy without changing what the adversary can do.

### 5.3 Block-RoPE permutation as a "free defense"

The "free" is misleading: the permutation is free of *online* cost
but introduces approximation error `e_C^attn`, eats some accuracy,
and protects against an attack class that doesn't exist in our
threat model. Skip.

### 5.4 vLLM-compatibility as an architectural goal

AloePri's killer feature for ByteDance — vLLM compatibility with
zero protocol-engine modification — is unattainable for us by
construction. Our protocol requires the TEE in the inner loop per
offload (mask + U-Verify + unmask). Trying to bend GELO toward
vLLM-compatibility would mean weakening our per-batch freshness
guarantee, which is exactly what makes our protocol open-weight-safe.

### 5.5 Replacing GELO with AloePri entirely

The temptation: AloePri reports <10% TPOT vs our 6–27%, and runs at
671B-scale vs our 0.6B–8B target. **Resist it.** AloePri achieves
those numbers by abandoning per-forward freshness, which is the
single property that gives GELO its open-weight-safe security
argument. Adopting AloePri-as-is means accepting that the protocol's
security collapses under public-weights deployment — exactly our
default.

## 6. Composability — Could AloePri Stack On Top of GELO?

In principle, two-layer composition is mathematically possible:

```
Client (in CVM):
   Apply AloePri token permutation τ — but τ has nothing to obfuscate
   here, because token IDs are TEE-internal in our design.

CVM TEE per forward pass:
   Apply GELO mask A_b (fresh per batch)

GPU:
   Computes on masked activations
```

But this composition gives nothing GELO doesn't already give. AloePri's
`τ` is a token-space permutation; our threat model doesn't have a
token-space leak (tokens never cross PCIe). AloePri's weight
obfuscation is irrelevant under public weights. The composition is
purely additive overhead.

**The only sensible composition is the empirical-validation one
described in §4: use AloePri's attack code against GELO's output
to validate our defense.**

§A.2's open question — *"per-prompt vs per-session rotation?"* — with
the resolved answer:

> AloePri's rotation is **per-deployment** (one `τ`, one `{P̂_i, Q̂_i}`,
> one `ε_e, ε_h`, fully baked into shipped weights). It is
> structurally a static-key scheme; security depends on the server
> not having access to the original weights. Under our openweight
> threat model, AloePri provides no security. See
> [`aloepri-vs-gelo.md`](aloepri-vs-gelo.md) for the full analysis.
> The only portable artifact is the empirical attack suite at
> github.com/sheng1feng/Aloepri/tree/main/src/security_qwen.
