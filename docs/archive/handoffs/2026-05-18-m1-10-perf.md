## 1. The bottleneck, decomposed

Latest run, `Qwen3-1.7B` on AMD RADV GFX1151 iGPU at n=2048 prefill +
16-token greedy decode (`m1-10-phase4-findings.md` §2.2 has the table):

```
cell                 TTFT      TPOT      total   vs plain
gpu_plain          6 443 ms    271 ms   10.8 s   (base)
gpu_gelo          72 904 ms    371 ms   78.8 s   +631 %
gpu_gelo_permuted 87 528 ms  1 063 ms  104.5 s   +870 %
```

The protocol path `decoder_block_cached` does **not** call
`offload_attention_*` for the Global attention step — attention is
computed in-TEE on CPU/BLIS per the locked M1.3 design decision
(`crates/gelo-embedder/src/decoder/forward.rs:340-371`). So the
**+631 % `gpu_gelo` overhead is not attention**. The
+870 % `gpu_gelo_permuted` overhead also isn't dominantly attention —
it's the F1+ permuted_cached path adding on top of the
linear-projection mask cost.

### 1.1 The dominant cost: GELO mask round-trip on the 6 linear-projection batches per layer × 28 layers

For each linear offload (`QKV`, `O`, `gate∥up`, `FfnDown`),
`InProcessTrustedExecutor` does:

```text
gelo:mask_apply   →  A · h_padded            CPU BLIS, (n+k, n+k) × (n+k, d_in)
engine:matmul     →  GPU does masked · W
gelo:mask_unapply →  Aᵀ · masked_out         CPU BLIS, (n+k, n+k) × (n+k, d_out)
```

At n=2048 with `k = shield_k = 8`:

| Offload | apply (d_in) | unapply count × d_out | FLOPs/layer |
|---|---|---|---|
| QKV (one mask) | 2 048 | 3 × {2 048, 1 024, 1 024} | ~51 GFLOPs |
| O | 2 048 | 1 × 2 048 | ~34 GFLOPs |
| gate ∥ up (one mask) | 2 048 | 2 × 6 144 | ~119 GFLOPs |
| FfnDown | 6 144 | 1 × 2 048 | ~68 GFLOPs |
| **Per-layer mask CPU cost** | | | **~272 GFLOPs** |

× 28 layers ≈ **7.6 TFLOPs of CPU BLIS matmul per prefill forward**,
none of which exists in `gpu_plain`. On the AMD Zen CPU at
~100-150 GFLOPs/sec multi-threaded BLIS this lands at **~50-75 s**,
matches the observed ~66 s delta exactly.

Plus one Haar QR sample per forward at `(n+k, n+k) = (2056, 2056)`:
~17 GFLOPs additional, ~0.5 s.

**This is what dominates long-context overhead. M1.10 was never going
to fix it.** It's a separate workstream — see §3 below.

### 1.2 The secondary cost (only on permuted_cached): K/V permutation copies + Gaussian noise

After the rayon-Gaussian fix (commit `cbea549`), the
`permuted_cached` TPOT delta over `gpu_gelo` at n=2048 is still
~692 ms (1 063 - 371). Breakdown estimate:

| Cost in `permuted_attention_cached` | Estimate/decode-step | Status |
|---|---|---|
| Gaussian noise on K (now rayon-parallel) | ~150-250 ms | **done — 1.59× win on TPOT** |
| **K, V permutation copies** | **~200-300 ms** | next hotspot (task #69) |
| Score-tensor PCIe round-trip | ~50-100 ms | unavoidable under F1+ |
| Mask + softmax + unpermute (in-TEE) | ~50 ms | not a target |
| Engine matmuls (Q·Kᵀ + probs·V) | ~50-100 ms | not a target |

K/V perm copies are now the single-threaded scalar memory-copy step
in `crates/gelo-protocol/src/attention.rs:permuted_attention_cached`
(approximately lines 326-340 — `for h in 0..num_heads { for (i,
&src) in perm_kv.iter().enumerate() { k_perm.slice_mut(...).assign(...); v_perm.slice_mut(...).assign(...); } }`).
At n_kv=2048: 16 × 2048 × 128 × 4 bytes × 2 tensors × 28 layers ≈
**900 MB of memory traffic per decode step**, all on one core.

## 2. Tried optimisations (with measured results)

### 2.1 F1+ resolution of the causal-mask leak (Phase 0 — landed `4da06aa`)

The original `permuted_attention` called `engine.softmax_batched` on
the masked score tensor. That tensor had `-∞` at blocked positions; the
engine could count exact zeros per row and recover π trivially. F1+
moves softmax in-TEE and replaces `-∞` with `-C = 30` so blocked
positions softmax to ~exp(-30) ≈ 1e-14 (non-zero at f32 precision).
**Three regression tests** in `crates/gelo-protocol/tests/permutation_attention.rs`
(`f1plus_*`) lock the property in.

Cost: one PCIe round-trip on the score tensor per call (~64 MB at
n=2048). The softmax CPU work is negligible at our shapes.

**Known residual**: the threshold-count attack (count probs entries
below ~1e-12 per row) still recovers π under F1+. Documented as a
follow-up (F1++ in security review §3); F1++ would add small Gaussian
noise on probs before sending back to the GPU. Not implemented; the
exact-zero attack — the load-bearing baseline — is closed.

### 2.2 Phase 1 dispatch — cached-path permuted attention (landed `735705c`, `1b94126`, `b3d8ed0`)

- New protocol primitive `permuted_attention_cached(q, k, v, scale, q_pos_offset, mask, cfg, rng)` handling asymmetric n_q ≤ n_kv with two independent permutations (π_q, π_kv).
- New trait method `TrustedExecutor::offload_attention_permuted_cached`. Default impl is plain asymmetric attention (PlaintextExecutor uses this as parity baseline); `InProcessTrustedExecutor` overrides to call the real protocol.
- New decoder wrapper `causal_gqa_attention_permuted_cached` in `crates/gelo-embedder/src/decoder/attention.rs`. Dispatch in `decoder_block_cached` routes Global-layer attention through it when `offload && cfg.perm_attention_enabled_for(n_q)`.
- Parity test `permuted_cached_dispatch_at_sigma_zero_matches_in_tee` (in `crates/gelo-embedder/tests/generation_harness.rs`) confirms byte-identical greedy tokens vs in-TEE at σ=0.

**Production default** (`cfg.use_perm_attention = false`) keeps today's behaviour. The new dispatch is opt-in. Cost when engaged → see §1.2 + §2.3.

### 2.3 Rayon-parallel Gaussian noise (`cbea549`)

`add_gaussian_3d_inplace` in `crates/gelo-protocol/src/attention.rs`
splits the heads axis above a 32 K-element threshold; each head gets
an independent ChaCha20 stream from a pre-derived seed; the parent RNG
is advanced by exactly `n_heads · 32 bytes` (deterministic-replay
contract preserved). Per-element noise distribution unchanged.

Measured win on `gpu_gelo_permuted` TPOT (n_kv=2048):
**1 693 ms → 1 063 ms (1.59×).**

**Why not the projected 10-30×:**
- Rayon work-stealing has ~100 μs fixed cost per call. 28 layers × 2 (Q, K) noise calls = 56 calls/decode-step → ~6 ms of pure scheduling overhead.
- Effective parallelism on this AMD CPU is ~8 cores, not 16.
- Gaussian sampling **inside each head is still scalar** — `rand_distr::StandardNormal::sample` is one Ziggurat call per element; no SIMD batching.

### 2.4 Other adjacent landed work (referenced for context, not retried this session)

- `gelo-protocol::out_attn_mult` — TwinShield 4-partition Q·Kᵀ
  outsourcing. Implementation matches the paper. Bench at n=2048 in
  the embedder path showed +14 ms/text setup overhead — **not engaged
  in the cached generation path** per the M1.3 design decision.
- `gelo-protocol::sim::offload_linear*` — paper-parity defaults via
  `with_seed`: per-forward Haar A + shield(8, 4.0); per-offload mode
  opt-in. This is the path producing the +631 % overhead at n=2048.

## 3. Deferred / structurally off-limits

### 3.1 On-GPU unmask (`Aᵀ` as GPU weight) — STRUCK 2026-05-18

Originally proposed in `docs/archive/prototype/inference-optimization.md` Tier
3.4. Would attack the §1.1 mask-round-trip cost directly. **Threat-
model-incompatible**: putting `A` on the GPU lets a hostile engine
compute `H = Aᵀ · masked` and recover plaintext H within one forward.
Also puts `A` on the PCIe bus, defeating TEE.Fail-class survival. Only
valid under confidential-GPU threat model (H100 CC), which GELO does
not target.

The doc was edited in commit `68cd468` to strike Tier 3.4 with a
threat-model-incompatible callout. **Do not re-propose without a
threat-model change.**

### 3.2 Fused-attention kernel (M1.10 Phase 2) — DEPRECATED 2026-05-18

The original M1.10 plan assumed a FlashAttention-style fused kernel
that takes `(q, k, v, scale, mask)`. Under F1+ the causal mask must
not be transmitted to the GPU; any fused kernel that accepts a mask
input re-introduces exactly the leak F1+ closes. Four work-arounds
all fail: (a) kernel infers causality from `q_pos_offset` — needs π
to apply the permuted-causal pattern, defeats hiding; (b) pre-noise
scores inside the kernel — research-level, no published recipe;
(c) HE-mask under encrypted softmax (TwinShield Liu '25) — we don't
carry HE; (d) pattern-invariant mask shapes — changes model semantics.

Documented in `docs/plans/m1-10-fused-permuted-attention.md` §5
Phase 2 and §6. Phase 4 bench (§2.2 of `m1-10-phase4-findings.md`)
empirically confirms: **attention compute is not the bottleneck**;
fused-flash would not have moved the long-context numbers.

### 3.3 Block-diagonal mask `A = diag(A_1, ..., A_B)` — deferred to security spike

Reduces mask matmul from O(n²·d) to O((n/B)·n·d). Privacy weakening
— cross-block correlations leak O(n/B) linear constraints per token.
Filed in `docs/dev/prototype/future-rnd.md` §5. Has the potential to
shave the §1.1 cost by a factor of B (typical B=4-8 → ~4-8× speedup
on linear mask work). **Not v1; needs a written security analysis
before adoption.**

### 3.4 HKDF-derived per-step mask material — deferred to security spike

At decode, deriving each step's A deterministically from
`HKDF(SessionKey, "gelo-llm.mask", step_idx)` amortises the per-step
Haar QR sample (currently ~17 GFLOPs/forward at n+k=2056) across the
session. Same gating as block-diagonal — needs a freshness-argument
write-up before adoption. Filed in
`docs/plans/m1-10-fused-permuted-attention.md` §10.

### 3.5 σ relaxation at Qwen3-1.7B shapes (M1.10.0.5)

Hidden No More tests σ=0.01 on GPT-2-class shapes. Our Qwen3-1.7B
shapes are different — σ might be tunable lower (cheaper noise) or
might need higher (more expensive). Open until someone runs a
recovery-attack bench on real Qwen3 activations. **Not in critical
path — gated on the perf items in §4 below landing first; otherwise
the noise sampler dominates the signal.**

### 3.6 End-to-end Q8 on Vulkan (task #60)

Burn-cubecl 0.20.1 doesn't expose stable quantised matmul on the
Vulkan path. Multi-week upstream / custom-WGSL effort. **Not a v1
item; revisit if quant matmul stabilises upstream.**

## 8. One non-obvious gotcha

The `qwen3_long_context_bench.rs` notes section explicitly says
"OutAttnMult auto-switch" — that was an earlier framing that was
**wrong**; the cached generation path bypasses OutAttnMult entirely
per M1.3. The bench note was corrected in commit `bbs9wzpp3` lineage
(see comments in the bench source) but **the misleading framing
persists in the §08 stat-strip subtitle of `docs/prototype/gelo-llm.html`**
and in older blog/PR descriptions. The correct framing: at n ≥ 2048
the dominant cost in `gpu_gelo` is **GELO mask round-trip on linear
projections**, not OutAttnMult engagement. Worth grepping the docs
for stale OutAttnMult-at-n=2048 claims and editing them out if found.
