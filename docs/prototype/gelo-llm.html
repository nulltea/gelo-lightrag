§01

Role in the project

where this fits

TODO — stub. Generation is the final stage that closes the private-RAG loop: embed the query, retrieve ciphertext candidates, rerank them inside the CVM, and now generate the answer in the same trust boundary rather than handing decrypted top-k chunks to a client-side LLM. The model weights are public; the protected asset is the user's prompt, the retrieved chunk text, and the generated tokens — all of which must stay confidential against a co-located untrusted GPU and an untrusted CVM operator. The same masking primitives the embedder validated carry over directly to autoregressive decoder forwards, with three new mechanisms: a Per-Layer Embedding table pinned to encrypted CVM DRAM, a hybrid attention router that keeps short-range sliding-window layers in-TEE while masking only the long-range global layers, and a KV-cache primitive that survives the per-batch mask rotation without re-permuting the cache every decode step.

§02

Definitions & glossary

terminology introduced by this stage

Only the terms specific to generation and to Gemma 4's architecture. Protocol terms (mask, shield, U-Verify, OutAttnMult, permuted attention) are defined in embedding §05 and gelo.md.

Term Short Meaning
Prefill — The single forward pass over the full prompt (system message + retrieved RAG context + user question) that populates the KV cache for all subsequent decode steps. Compute-intensive, attention-dominated past ~2k tokens; the "long" workload of the two phases.
Decode — Per-token autoregressive step. The new token's query attends to all cached K, V; one new K/V pair is appended. Per-step compute is small, dispatch-bound; the "narrow" workload of the two phases.
Time-to-first-token TTFT Latency from request arrival to the first output token. Dominated by prefill cost. The user-visible "is the model alive yet" number.
Time-per-output-token TPOT Steady-state inter-token latency once generation starts. Dominated by decode cost. The user-visible "how fast does it read" number.
KV cache — The materialised (K, V) pair for every previously-seen position, per layer, per head. Lives in encrypted CVM DRAM; sized 2 · n_layers · n_kv_heads · n_cache · d_head. ~440 MB for Qwen3-0.6B at n_cache=4k; ~2 GB-class for Gemma E4B at long context. The heavy state of the decode loop.
Per-Layer Embedding PLE An additional embedding table introduced by Gemma 3n / Gemma 4 E-variants, shape [vocab × n_layers × d_ple] (262 144 × 35 × 256 for E4B), indexed by token_id per layer rather than once at input. Address-bus-leaking under our threat model — token-id-keyed gathers on a GPU reveal the prompt in plaintext with no inversion needed. The fix is structural: the table lives in CVM DRAM; the TEE performs the gather; only the public projection matrix that lifts the gathered row to d_hidden ever sees GPU.
Hybrid attention — The Gemma 3 / Gemma 4 design choice that interleaves cheap local layers (sliding-window causal attention, window W tokens) with full-context global layers. E2B uses 4:1 (four local : one global); E4B uses 5:1; the last layer is always global. Cuts per-token attention FLOPs ~3.86× at n=8k on E4B and shifts the bandwidth bottleneck onto the small minority of global layers — the only ones that need GPU offload at long context.
Sliding-window attention SWA Causal attention restricted to the last W tokens (W = 512 on small Gemma 4, W = 1024 on 31B). Reduces attention from O(n²) to O(n·W). In-TEE-tractable at all our context lengths; does not compose with permuted attention because the band-diagonal mask is not permutation-invariant.
p-RoPE — Proportional RoPE. Rotation applied to the first p · d_head dimensions only; the remaining (1−p) · d_head dims pass through unrotated. Gemma 4 global layers use p = 0.25. One-line change to the existing RoPE kernel — defaults to full-rotation (p = 1) when not set.
K = V tying — Gemma 4 global layers materialise K and V as the same tensor (the "K equals V" trick). Halves global-layer KV-cache memory and lets the protocol sample one mask, one permutation, or one OutAttnMult partition for both — not two. Local layers keep separate K, V.
Grouped-query attention GQA Q has more heads than K/V; multiple Q heads share one K/V head. Gemma 4 uses 8:1 (eight Q heads per KV head). Already handled unchanged by the Qwen3 attention path in the embedder/reranker; reused here.
MatFormer — Nested-architecture training that produces the E2B model as a structural subset of the E4B model. Nominally lets a single weight blob serve both sizes; in practice Gemma 4's E2B and E4B have different attention ratios (4:1 vs 5:1), so the prototype treats them as two artifacts with two loaders.
Address-bus leak — An attack class where an observer on the memory or PCIe bus learns content from which addresses are accessed, not what is read at them. Defeats mask-based confidentiality on table lookups (PLE, MoE expert selection, embedding gather) because the index, not the value, is the secret. Mitigation here: every secret-indexed table read happens in-TEE; only public-key projections cross PCIe. Empirically validated by the TEE.Fail DDR5 bus-interposer result (Oct 2025), which is precisely the threat the per-batch-fresh-mask architecture is designed to survive — even if mask state is later extracted offline, no live forward pass is recoverable.
Streaming output — Per-token emission over the open RATLS connection as soon as each token is sampled, rather than batched-at-EOS. Surfaces TPOT to the user but exposes per-token timing on the wire — covered in §03.
SCX KV-cache encoding SCX Stateless per-user KV-cache encoding scheme (Yuan et al., SIGCOMM 2025) that derives the encoding key from (session_id, layer_id, position) at write time and reads the encoded K, V back without per-step re-permutation. Candidate decode-phase primitive; complements permuted attention on prefill. Not yet adopted; evaluated against the threat model when the decode bench lands.
§03

Threat model

what each component sees · what it cannot · what's new vs embedding/rerank

Same trust posture as embedding and reranking: hardware-attested TEE on CPU, information-theoretically blind GPU, everything else hostile. The openweight assumption holds — Gemma 4 weights pin to a public HuggingFace revision; the SHA-256 of the loaded bytes rides as model_identity through the attestation report. Generation extends the model with three new exit channels (PLE gather, KV cache, per-token streaming); each is closed structurally rather than statistically.

Six exit-channel leaks specific to generation, each closed by construction:

Leak How it's closed
PLE address-bus leak (P0) The full [262 144 × n_layers × 256] int8 PLE table is provisioned into the InProcessTrustedExecutor's encrypted CVM DRAM at model load. Per-token row selection (PLE[token_id, layer_idx]) happens inside the TEE; only the resulting (n, d_ple) tensor — projected up to d_hidden via the public per-layer projection matrix — ever crosses PCIe, and it crosses under the same per-batch mask A as the rest of the activation stream. A PCIe observer sees no token-id-keyed gather pattern. Verified by tests/ple_pcie_leak.rs, an attacker-simulator that asserts zero PLE-side reads on the offload trace.
KV-cache content leak The KV cache for both local and global layers lives entirely in encrypted CVM DRAM, indexed by (layer_idx, head_idx, position). The GPU never touches KV bytes — attention's Q · Kᵀ and P · V for in-TEE local layers run on CPU under AOCL-BLIS; for global layers the per-step Q goes out under mask, but K and V are pulled from cache, masked freshly, and never persisted GPU-side. Decode-phase π handling — fresh π each step versus session-persistent π — is the decision point that gates SCX-style encoded-KV adoption; see §08.
Per-token TLS timing Streaming output flushes one ciphertext frame per sampled token over the RATLS connection. An on-path observer sees per-token timing; combined with public knowledge of the GELO+Gemma forward-pass cost, this is mild side information about token entropy (cache-friendly tokens decode faster). Mitigations available at deployment time: (a) emit at a fixed cadence padded by sleep; (b) batch tokens into fixed-size frames at the cost of latency; (c) disable streaming entirely and return at EOS. The protocol surface is unchanged by the choice.
Generation-length leak Total token count is observable from TLS frame count and connection duration. Treated as out-of-scope at the protocol layer — same posture as the existing reranker's k_max being a function of the deployment, not the request. Fixed-length deployments (always emit exactly n_max tokens, padded with EOS-after-stop) are available as a wrapper around the same generation loop.
Sampler-state leak Sampling (greedy, top-k, top-p, temperature) runs entirely in-TEE on plaintext logits. The RNG is a ChaCha20 seeded from the per-session SessionKey; the GPU sees neither the logits nor the sample. Logit values are zeroized after sampling; only the sampled token_id survives to the next step and to the KV-cache append.
Tied-LM-head dot-product leak The output projection is H_final · token_embeddingᵀ over the 262 144-row vocabulary. The mask GEMM at this shape is unchanged from the reranker's tied-LM-head pattern — masked offload of the same form as every other linear. The vocab-row gather that follows sampling (next-step input embedding) is again token-id-keyed and stays in-TEE for the same reason PLE does.

Not covered. Total prompt length and total generation length are visible on the wire and are treated as request metadata. Side channels in the TEE itself (cache timing, branch prediction) are the responsibility of the SEV-SNP hardware envelope. Whether a deployment chooses to pad-and-batch tokens to obscure per-token timing is a deployment policy, not a protocol property.

Validating result. The TEE.Fail DDR5 bus-interposer attack (Oct 2025) defeats SGX, TDX, and SEV-SNP key sealing in the offline case and uses extracted CPU-TEE keys to compromise H100 confidential GPU. The architectural choice that survives it is the protocol's per-batch fresh Haar mask: mask material exists only for the duration of one forward pass; an offline TEE compromise reveals nothing about any past forward and nothing about any future one until a fresh mask is sampled. Generation inherits this property unchanged — every prefill is one forward pass, every decode step is one forward pass.

§04

Supported workloads

the two phases of autoregressive generation · what protocol covers each

Generation has two phases with structurally different compute shapes and structurally different cost profiles. The protocol covers both, but with different attention paths and different cost trade-offs. The runner exposes them as a single generate(prompt, max_tokens) entry point that does prefill once and then loops decode.

Phase Shape Dominant cost Attention path (local layers) Attention path (global layers) Mask cadence
Prefill n_q = n_prompt (full prompt) attention O(n²) at long context · linear projections at short context in-TEE causal sliding-window (W = 512) · CPU under AOCL-BLIS length-based auto-switch: in-TEE for short n · fused permuted attention (M1.10) on the GPU past threshold · OutAttnMult is the 3-dispatch fallback before M1.10 lands one Haar A for the whole prefill forward
Decode n_q = 1, n_kv = n_cache (growing) dispatch overhead · KV-cache bandwidth · linear projections in-TEE causal sliding-window · trivially small in-TEE always. Per-step attention math is microseconds-scale at n_q = 1; the auto-switch threshold doesn't engage on decode. KV-cache bandwidth (orthogonal axis) is what SCX in §08 optimises. one fresh Haar A per decode step (per-token)

Why prefill and decode want different attention paths. Prefill at n = 4k materialises a (heads, n, n) score tensor of ~1 GB per layer that the 3-dispatch permuted-attention path forces through device memory three times — bandwidth-bound to ~3.2 GB/layer × 28 layers ≈ 90 GB on the integrated-GPU dev box (~1.8 s pure traffic per prefill). The FlashAttention-style fused permuted kernel (M1.10) drops that to ~130 MB/layer and lands the workload within ~2× of unprotected baseline. Decode at n_q = 1 never materialises a square score tensor — per-step attention compute is microseconds and stays in-TEE; the bottleneck instead is the KV-cache bandwidth read at every decode step, addressed by the SCX-class encoded-KV primitive (§08) rather than by any attention-dispatch change.

Why hybrid attention matters so much here. Gemma 4 E4B's 5:1 ratio means 35 of 42 layers are local-sliding-window. At n = 8k with W = 512 those 35 layers cost ~3.86× less attention compute than a dense 42-layer model, and they stay in-TEE comfortably because the per-layer attention is now O(n · W) rather than O(n²) — no offload, no mask round-trip, no PCIe bandwidth. Only the seven global layers need the permuted/OutAttnMult path. That ratio is what makes Gemma 4 a particularly good fit for this protocol: most of the attention work disappears into the TEE-cheap regime, and the protocol-expensive path engages on only the layers that actually need long-range context.

Sampling. Greedy and top-p / top-k / temperature sampling all run in-TEE on plaintext logits. The seed is derived from the per-session SessionKey; deterministic replay (same prompt, same session) is possible at temperature = 0 for debugging and reproducibility benches without breaking confidentiality of the live request.

§05

Supported model architectures

Qwen3 (v1 demonstrator) and Gemma 4 (Phase 1.5 architecture target)

Two model families, two roles. Qwen3-1.7B is the current v1 demonstrator — a vanilla GQA decoder that drops cleanly into the existing protocol surface and runs real-weight greedy generation end-to-end today. Gemma 4 E2B/E4B remain the Phase 1.5 architecture target — they exercise every novel primitive in the spec (PLE in TEE DRAM, hybrid local/global attention, K=V global layers, p-RoPE, GeGLU, AltUp, cross-layer KV sharing), but require ~4-5 weeks of decoder refactor work before the first real-weight forward pass — see gemma4-architecture-roadmap. The SHA-256 of backbone bytes + (optional) PLE table bytes + LM-head bytes + per-layer attention-class vector + tokenizer rides as model_identity through every attestation report — a tokenizer drift or a re-numbered <eos> trips the attestation regardless of family.

Variant Layers Hidden · Inter · d_head Q · KV heads Attention RoPE θ PLE Status
Qwen3-1.7B 28 2048 · 6144 · 128 16 · 8 full causal 1 M — v1 demonstrator
Qwen3-4B 36 2560 · 9728 · 128 32 · 8 full causal 1 M — v1 stretch (same code path)
Gemma 4 E2B 35 1536 · 6144 · 256 / 512 8 · 1 4:1 local/global · W=512 10 K / 1 M 262 144 × 35 × 256 · int8 Phase 1.5 target
Gemma 4 E4B 42 2560 · 10 240 · 256 / 512 8 · 2 5:1 local/global · W=512 10 K / 1 M 262 144 × 42 × 256 · int8 Phase 1.5 scaling target

Qwen3-specific addition: QK-norm. Qwen3 introduces per-head RMSNorm on Q and K before RoPE (self_attn.q_norm.weight / k_norm.weight, each (head_dim,)) — the only Qwen3-vs-Qwen2 architectural delta. It fits as one optional step between the QKV projection and the rotary embedding; no structural refactor was needed. The loader reads the tensors when present (Qwen3 family) and skips the step when absent (Qwen2 / LLaMA / Mistral / Qwen3-Embedding paths stay byte-identical to their pre-Qwen3 behaviour).

Qwen3.5 was investigated and is out of scope. The family ships as Qwen3_5ForConditionalGeneration — a multimodal VLM with a 24-layer ViT, image / video tokens, and a text backbone that replaces transformer attention with GDN-style linear_attention (Mamba/SSM) layers in a 3:1 hybrid with full_attention, MRoPE (interleaved mrope_section), partial_rotary_factor: 0.25, an MTP head, and attn_output_gate. Strictly harder than Gemma 4, not easier. Plain Qwen3 is the only Qwen-family vanilla decoder option.

Gemma 4 31B dense is out of v1 scope (2026-05-18 decision). The architecture (5:1 hybrid, W = 1024, no PLE, ~5120 hidden) would fit the protocol primitives unchanged, but the ~31 GB weight footprint forces a 64 GB SEV-SNP SKU. Revisit when such a SKU becomes available; until then E2B and E4B are the v1 surface.

Why Qwen3-1.7B is the v1 demonstrator

Vanilla Qwen3ForCausalLM — full causal attention, single RoPE table, SwiGLU, tied embeddings, no PLE, no hybrid attention, no p-RoPE, no softcap, no AltUp. The single Qwen3-specific addition (per-head QK-norm before RoPE) is an optional pre-rotary step that fits cleanly under the existing offload + RoPE plumbing. ~3.4 GB bf16 footprint downloads from Qwen/Qwen3-1.7B on first run. The same DecoderWeights pipeline that loads Qwen3-Embedding-0.6B services this model — exercises the full GELO protocol surface (mask · shield · OutAttnMult auto-switch · per-batch fresh A) on real-weight generation without blocking on Gemma-only architectural work.

Why Gemma 4 is the Phase 1.5 architecture target

Smallest family member (E2B) that exercises every novel protocol surface in the spec — PLE in TEE DRAM, hybrid local/global attention, K=V global layers, p-RoPE, GeGLU activation, AltUp, cross-layer KV sharing. ~4.5 GB total weight footprint fits a 32 GB CVM SKU. E2B → E4B is the natural scaling axis once the architecture work lands. The Gemma 4 forward path is ~4-5 weeks of decoder refactor (per-class head_dim 256/512, per-class rope_theta 10K/1M, GeGLU dispatch, cross-layer KV sharing, AltUp). Tracked in gemma4-architecture-roadmap.

MatFormer is intentionally not exploited. Gemma 4 nominally lets E2B share weights with E4B via nested training; in practice the attention ratios differ (4:1 vs 5:1), the layer counts differ (35 vs 42), and the hidden sizes differ. The prototype treats E2B and E4B as two separate artifacts with two loaders, sharing the same forward-pass code path. A shared weight blob would save disk but doesn't reduce runtime memory (only one variant is loaded per CVM) and complicates the attestation binding.

Gemma 3n and Llama / MoE paths. Gemma 3n (an earlier PLE-using model) is supportable on this stack with the same PLE-in-TEE-DRAM mechanism but is not a v1 target — superseded by Gemma 4 E2B. Llama dense is already supported in the embedder/reranker code path and trivially extensible (no PLE, no hybrid attention, no p-RoPE, no QK-norm). MoE variants (Qwen3-MoE, DeepSeek-V2) are deferred — the routing histogram leak is a separate protocol surface (CryptoMoE balanced-dispatch primitive) and out of scope for this stage.

Compute flow & trust boundaries

prefill once · decode loop · per-layer block detail

Two figures trace one Gemma 4 decoder block under each phase. FIG. 03a covers prefill (full prompt, attention O(n²) on global layers). FIG. 03b covers one decode step (n_q = 1, growing KV cache, attention O(n_cache · d_head)). Both use the same per-block topology — what differs is the attention input shape, the KV-cache access pattern, and the PLE gather length (full prompt for prefill, one token for decode).

FIG. 03a — Gemma 4 prefill · one decoder block · pre-LN · SwiGLU · GQA(8:1) · hybrid attention (local-SWA + global) · PLE gather in-TEE · rev 2 · 2026-05-18 · 4 GPU GEMMs per block blue arc = residual · solid red = TEE flow · dashed amber = masked PCIe transit (mix → / unmix ←) · green = PLE gather (TEE-local)
FIG. 03b — Gemma 4 decode · one step · n_q = 1 · n_kv = n_cache · same block topology · attention reads from CVM-resident KV cache · rev 2 · 2026-05-18 · 4 GPU GEMMs per block same legend · one fresh Haar A per decode step · π handling decided at §08

What to look for between the two figures. ① Per-block topology is identical — both phases dispatch 2 matmul_many + 2 matmul + 1 attention per block, repeated × 35 (E2B) or × 42 (E4B). What differs is the input shape of every GEMM: (n_prompt, d_hidden) in prefill, (1, d_hidden) in decode. The protocol does the same work; the wall-clock cost regime is what flips. ② PLE gather happens in both phases, on different shapes: n_prompt rows in prefill, one row per decode step. Both are TEE-only — green border on the box, never crosses PCIe. ③ KV cache is written by prefill (full prompt at once) and appended by decode (one position per step). Lives in encrypted CVM DRAM in both phases. ④ Attention dispatch diverges by layer class, not by phase. Local layers stay in-TEE in both phases (CPU under AOCL-BLIS); global layers cross PCIe under mask in both phases. ⑤ Dispatch overhead dominates decode (every block's four (1 × d) · (d × d) GEMMs and one Q · Kᵀ_cache cache-read sum to a tight per-token budget); attention compute dominates prefill at long context (a single global-layer Q · Kᵀ is the largest GEMM in the whole stack).

Mask cadence in the generation loop

The protocol defaults from embedding and reranking carry over unchanged: per-forward-pass Haar mask sampling with shield rows (k = 8 at energy 4·mean‖h‖), and per-offload shield freshness. The new dimension is generation:

  • Prefill is one forward pass — one Haar QR, then the same A rides every masked offload in the block stack. Same cost amortisation as the embedder.
  • Each decode step is one forward pass — one fresh Haar QR per token. At d_hidden = 2560 (E4B), the QR is the dominant per-step protocol cost; HKDF-derived mask material (§08 lever) can amortise it across steps for free if the security analysis confirms equivalence.
  • π in permuted attention is the open decode-phase decision: fresh π per step requires SCX-style encoded-KV (§08); session-persistent π enables in-place cache reuse but trades one weaker decode-step argument. Documented in §08, gated on the security spike when the bench harness lands.

Why per-forward + shield remains the right default. Per-offload Haar sampling would cost ~96–140 QRs per forward at Gemma 4 shapes; at d = 2560 each QR is ~20–30 ms, so per-offload mask sampling alone would dominate every other cost in the system. Per-forward + shield is the paper's §3.2 + §4.2 trade and what the executor ships by default — see reranking §05 for the per-cadence breakdown that motivates it.

Deferred to follow-up rounds

  • SCX-style KV-cache encoding for decode. The decode-phase KV-cache bandwidth grows with context length and dominates per-token cost past ~4k tokens. SCX (Yuan et al., SIGCOMM 2025) derives the per-position encoding from (session_id, layer_id, position) at write time and reads encoded K, V back without per-step re-permutation — the candidate primitive for amortising the cache-bandwidth cost. Security analysis against Hidden-No-More / KV-Cloak adversaries is the gating spike before adoption. This is the only decode-phase optimisation that addresses cache bandwidth at long context; attention compute at decode stays in-TEE always.
  • HKDF-derived mask material for amortised decode-step QR. If decode-step Haar QR becomes the dominant per-token cost at E4B scale, deriving each step's mask deterministically from HKDF(SessionKey, "gelo-llm.mask", step_idx) amortises the QR across the session under a freshness argument tied to the session secret. Lever; not on critical path; landing condition is the E4B bench showing mask-sample > ~10% of TPOT.
  • Speculative decoding. Many production LLM stacks use draft-model + verify-step patterns for throughput. The protocol implications under per-batch fresh A are completely unexplored.
  • MoE generation. Qwen3-MoE / DeepSeek-V2 / Gemma 4 MoE 26B-A4B add a router-histogram leak that's orthogonal to everything in this stage; the right protocol surface is CryptoMoE-style balanced dispatch, which is a separate ~3–6 month workstream. Out of scope here.
  • Score-DP / token-DP accountant. Formal (ε, δ) budget over emitted tokens. Same trigger as the reranker's score-DP item: only relevant if a deployment needs to export per-token probabilities for downstream calibration.
  • ECDH-bound session-key handshake. Currently session_secret is supplied per request. The reranker's M5.9 swap-in (attestation-bound ECDH KEX with the SessionKey::derive API unchanged) applies here verbatim.
  • Gemma 4 31B dense. Dropped from v1 scope (2026-05-18). 5:1 hybrid, W = 1024, no PLE, ~5120 hidden — same protocol primitives apply, but the ~31 GB weight footprint forces a 64 GB SEV-SNP SKU. Revisit when such a SKU is available.

Open decisions

  • Decode-phase π (gates SCX adoption, not v1): persistent across a generation session, or per-step refresh? Per-step requires SCX-class encoded-KV; persistent requires a fresh security argument against ARROWMATCH / Hidden-No-More extensions to autoregressive settings. v1 doesn't use permuted attention at decode (auto-switch keeps it in-TEE), so this is purely a §7.1 follow-up question.
  • PLE table dtype: int8 vs fp16. int8 saves ~700 MB at small quality loss; fp16 is bit-exact. Default int8 unless M1.8 accuracy validation flags a benchmark moving by > 0.5pp.
  • Streaming vs batched emission: deployment-level choice. Streaming exposes per-token timing; batched hides timing but loses TPOT to the user. Both wired; default streaming.
  • Fused permuted attention option (Option A vs B vs C in M1.10): decided at start of M1.10 based on cubek-attention API stability and burn-cubecl upstream state at that moment.

Decisions locked (2026-05-18)
