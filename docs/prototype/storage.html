§01

Role in the project

where this fits

Once the embedding layer has produced a pooled unit vector, the storage layer answers two questions: what gets persisted and what does the operator see when a query arrives? The literature has two practically-distinct answers; this prototype ships both as parallel services and lets the deployment pick at the type level.

GeloRAG (CAPRISE-at-rest) — the index server holds distance-preserving ciphertext vectors. Cosine ANN runs directly on the ciphertext; decryption only happens inside the TEE (or on the client, in alternative key models). The cloud operator with full DB access learns ciphertext geometry and access patterns but not the documents themselves. The current SEV-SNP runner ships this path with Option 3 key management (two-party HKDF — spec).

RemoteRAG (planar-Laplace + Paillier rerank) — the index server holds plaintext doc embeddings (chunk text still AES-encrypted). The query gets formal (n, ε)-DistanceDP via planar-Laplace noise; a second-stage Paillier homomorphic dot-product reranks against the clean query, recovering ~100% recall. The cloud operator with full DB access can read the doc embeddings but cannot learn what was queried.

The two paths protect different things and trade off opposite directions: CAPRISE keeps embeddings encrypted at rest but leaks distance ordering; RemoteRAG keeps embeddings plaintext at rest but gives the query a formal DP guarantee. They cannot be stacked: Paillier's additive homomorphism needs plaintext doc embeddings as exponents, and CAPRISE's s·e + r·u form destroys that structure.

§02

Threat model

CAPRISE vs RemoteRAG · pick on what you're protecting

Both paths share the same trust-zone model — client + TEE trusted, GPU + storage + network untrusted. They differ on what an untrusted-storage adversary actually recovers from full DB access.

Axis GeloRAG · CAPRISE-at-rest RemoteRAG · plaintext-doc + PHE rerank
Primitive Property-preserving scale + radial-Gaussian noise on the unit sphere, keyed via PRF IND-CPA additively-homomorphic public-key encryption (Paillier) + planar-Laplace noise on query
Server-side index CAPRISE-encrypted ciphertext vectors · same dimensionality as plaintext Plaintext doc embeddings (chunk text still AES-encrypted)
Server-side compute Plain cosine over ciphertext — that's the point of "distance-preserving" HNSW DistCosine over plaintext doc embeddings + Paillier dot-product rerank

What it does NOT learn Plaintext docs · plaintext queries · which plaintext vector each ciphertext encodes Plaintext docs · clean query q · the dot-product ⟨q, e_d⟩ (Paillier-encrypted)
Formal guarantee None — distance-preserving by construction leaks geometry. Defensible only as a key-rotation / data-residency control. (n, ε)-DistanceDP on queries; n = embedding dim (not cluster count — common misreading)
Failure mode if key-holder is compromised Vec2Text-style text reconstruction on the recovered embedding (mitigated by composing with DP-Forward at the embedder) DP budget exhausts after enough queries; doc-side breach exposes plaintext doc embeddings

Ciphertext size 1 × n floats (same as plaintext) n × ~256 B per Paillier ciphertext (one per dim)
Composes with DP-Forward ✓ at the embedder, pre-encryption — survives CAPRISE decryption ✓ on the doc side at ingest, or on the query alongside planar-Laplace

Picker. Curious server + trusted key-holder + want fast retrieval → CAPRISE. Untrusted retrieval server + willing to trade query latency for formal DP + doc embeddings are not the secret you're protecting → RemoteRAG.

AES-256-GCM authenticated encryption of document chunk text. Used by both storage paths. Key under Option 3 is HKDF-derived from user_x_sk ‖ tee_user_x_sk; under RemoteRAG it's client-generated.

raw chunk text returned alongside top-k embeddings
threat
passive adversary with index access; GCM tag catches mutation
cost
negligible — bandwidth-bound; AES-NI ≫ network
shared · DP layer

DP-Forwarddp_forward

Analytic Matrix Gaussian Mechanism — irreversible Gaussian noise on the pooled embedding for formal (ε, δ)-Sequence-LDP. Survives any downstream encryption.

method
L2 clip · Balle–Wang analytic σ · isotropic N(0, σ²I). DpForwardConfig::config_digest() folds into model_identity so the SEV-SNP report commits to (ε, δ, C, σ).
source
Yue et al. · arXiv 2309.06746 · CCS 2023
secures
the residual after a key-holder decrypts the CAPRISE index (defence-in-depth); doc-side noise at RemoteRAG ingest
threat
compromised CAPRISE key / TEE breach / legal compulsion / key-rotation bug
cost
µs per pooled vector · destroys NFCorpus retrieval at ε=4 — kept callable, off by default
GeloRAG · at-rest cipher

CAPRISErag_core :: caprise

Distance-preserving embedding encryption — server runs plain cosine over ciphertext, recovers cleartext only with the key. Asymmetric noise coefficients keep query-doc rank order intact.

method
scale + radial-Gaussian noise on the unit sphere · PRF-derived per-nonce direction · 3/8 coefficient for docs, 1/8 for queries
source
Ye et al. 2026 / prRAG — arXiv. Built with seeded path CapriseKey::from_seed for HKDF-derived material.
secures
stored embedding vectors against a curious operator without the key
threat
curious cloud operator with full DB access. NOT secure against a key-holder running Vec2Text — that's what DP-Forward catches.
cost
~0 retrieval overhead · 2,339 vec/s encryption · Vec2Text BLEU drops 83 → 12.4 (paper)

RemoteRAG · query DP

Planar-Laplaceremote_rag :: planar_laplace

Query-side (n, ε)-DistanceDP — perturb q to q′ = q + r·v. r ~ Gamma(n, 1/ε), v uniform on S^{n−1}. The server cannot distinguish the true query within radius n/ε.

method
Gamma-distributed radius · Gaussian-normalised direction. n is embedding dim, not cluster count — calibration is ε ∼ 10n…50n.
source
RemoteRAG — Cheng et al. · ACL Findings 2025 · arXiv 2412.12775
secures
the query operand on its way to a plaintext-index server. Stage-2 Paillier rerank recovers exact ranking under the clean query.
threat
untrusted retrieval server. Mutually exclusive with CAPRISE-at-rest.
cost
µs per query · latency budget lives in Paillier (Stage 2)
RemoteRAG · PHE rerank

Paillier rerankremote_rag :: paillier

Additively-homomorphic dot-product. ∏ᵢ Enc(qᵢ)^{e_d[i]_int} = Enc(⟨q, e_d⟩). Stage 2: reranks Stage-1 candidates against the clean query without revealing it.

method
Custom Paillier on num-bigint (pure-Rust, no GMP). Full CRT factor precompute · fixed-point quantisation · signed-result decode · rayon hot path.
source
RemoteRAG protocol. Algorithms adapted from fast-paillier 0.3.2 (LFDT-Lockness, Apache/MIT). Pure-Rust backend, no LGPL footgun.
secures
exact query–doc cosine on a plaintext-index server without revealing the query
threat
untrusted retrieval server
cost
~750 ms CRT-encrypt + ~47 ms × k′ rerank @ 256-bit · HNSW Stage 1 → 23.5 ms end-to-end @ 10k docs

§04

Compute flow & trust boundaries

two distinct shapes · same trust zones

Both paths share the embedding step. They diverge at "what happens to the pooled vector"; they share the AES-encrypted chunk-payload mechanism throughout.

FIG. 03 — GeloRAG · CAPRISE-at-rest with two-party HKDF · query path solid red = trusted secret · solid teal = ciphertext over wire · dashed = boundary
FIG. 04 — RemoteRAG · planar-Laplace + Paillier rerank · query path solid plum = Paillier homomorphic transit · solid red = trusted secret

GeloRAG steps (FIG. 03)

  • ① Client opens RATLS, verifies attestation, sends {tenant_id, user_x_sk, text}
  • ② TEE embeds via GELO mask
  • ③ TEE looks up (or creates) tee_user_x_sk[tenant], runs HkdfPolicy::derive
  • ④ TEE CAPRISE-encrypts the query with the 1/8 coefficient
  • ⑤ Storage server runs cosine ANN over CAPRISE ciphertext, returns top-k′
  • ⑥ TEE CAPRISE-decrypts + AES-decrypts; zeroizes derived keys
  • ⑦ Client gets plaintext ranked top-k over the same RATLS session

RemoteRAG steps (FIG. 04)

  • ① Client embeds (locally or via TEE) → q; samples r ~ Γ(n, 1/ε), v ~ U(S^{n−1}); computes q′ = q + r·v and enc_q = Paillier(q)
  • ② Client holds the Paillier sk; only pk is sent to the server (at ingest)
  • ③ Server runs Stage-1 HNSW DistCosine on plaintext doc embeddings using the noisy q′ → top-k′ candidates
  • ④ Server runs Stage-2 Paillier rerank: ∏ᵢ enc_q[i]^{e_d[i]_int} per candidate
  • ⑤ Server returns Paillier ciphertexts + AES chunk blobs
  • ⑥ Client decrypts each ⟨q, e_d⟩ with sk, sorts to true top-k, AES-decrypts chunk text

GeloRagTwoPartyService — what the runner binds

  • new(embedder, verifier) · with_params(...) — no scheme field; derived per request
  • ingest_chunks_for(tenant_id, user_x_sk, chunks) — HKDF · CAPRISE · AES · index, zeroize at return
  • query_for(tenant_id, user_x_sk, text, top_k) — re-derive same keys deterministically · search · decrypt
  • rotate_tenant(...) — stub; runner returns 501. Full re-key flow in M8.
  • forget_tenant(tenant_id) · tenant_known · scheme_identity() → [u8;32]
  • TwoPartyError::UnknownTenant → HTTP 410 Gone (loud failure on CVM restart)

RemoteRagService — what the parallel service binds

  • new(embedder, planar_laplace_cfg) · with_paillier_bits(256) · with_over_fetch_factor(3) · with_seed([..32])
  • ingest_chunks(chunks) — embed → AES-chunk-encrypt → index plaintext embedding + ciphertext chunk
  • query(text, top_k) — Stage 1 planar-Laplace ANN → Stage 2 Paillier rerank → return Paillier ciphertexts
  • Internal: IndexEntry { chunk_ct, embedding: Vec<f32> } · HNSW DistCosine ≥ 256 docs
  • Client holds Paillier sk + chunk AES key; private key NEVER reaches the server

Runner wiring—RemoteRAG is a parallel service today; the SEV-SNP runner ships the CAPRISE path. Wiring a second route is straightforward when needed.

Open levers (Storage layer)

  • HNSW swap in InMemoryEncryptedIndex — sub-ms search at 100k+ encrypted docs. RemoteRAG already does this for plaintext docs.
  • Variant-B persistence for tee_user_x_sk — KMS-released disk-sealed secret, survives CVM restart at same MEASUREMENT. M7.
  • Rotation flow — /rotate decrypts-under-old + re-encrypts-under-new atomically. M8.
  • Server-side ZK proof of correct ANN — V3DB-style. Open research gap; would close the integrity story for the storage side analogous to U-Verify on the GPU side.

Out of scope (explicitly)

  • Encrypted-everywhere ANN — would require FHE on the doc-side embedding while preserving Paillier composability. No practical scheme.
  • Cross-encoder rerank server-side — running a transformer under FHE is not practical. Client-side cross-encoder rerank on the recovered plaintext top-k is the design intent.
  • Stacking CAPRISE + RemoteRAG — mutually exclusive by construction. Documented in docs/prototype/remote-rag.md.
