# CAPRISE-in-TEE with Two-Party KDF (Plan + Delivery)

> **Scope.** Implementation plan for wiring CAPRISE encryption inside the
> SEV-SNP CVM under design-doc **Option 3 — Two-Party KDF (HKDF)** from
> `docs/research/private-rag-system-design.md` §4 storage. The CVM derives
> the CAPRISE key from one client-held secret + one TEE-sealed secret, then
> immediately zeroizes the client half. Forward-secure against TEE-only
> compromise. Covers both the ingest and query flows.
>
> **Status — delivered 2026-05-14.** M1–M6 landed. The runner today wires
> `GeloRagTwoPartyService` (CAPRISE keys derived per-request from HKDF);
> the previous `IdentityScheme` pass-through is gone. 150 / 0 / 31
> passed/failed/ignored across the workspace test suite. See §10 for the
> per-milestone delivery table with file references; M7 (KMS-released
> persistence) and M8 (rotation flow) remain deferred.
>
> **Companions.** `gelo.md` for the GELO/SEV-SNP substrate · `dp-forward.md`
> for the noise layer that composes with CAPRISE · the design doc Option 3
> sub-section in `docs/research/private-rag-system-design.md`.

---

## 0. Definitions

- **`user_x_sk`** — 32-byte client-held tenant secret. Provisioned once
  per tenant on the client; transmitted to the TEE only over an attested
  RATLS session; **never persisted in the enclave**.
- **`tee_user_x_sk`** — 32-byte TEE-held tenant secret. Generated by the
  CVM on first contact with a tenant id and bound to the attested code
  identity. Persisted (with constraints — see §5).
- **`caprise_seed_key`** — the 32-byte seed used by
  `CapriseKey::seed_key`; controls the PRF-derived direction + radius
  noise on every encrypted vector.
- **`aes_chunk_key`** — 32-byte AES-256-GCM key used by
  `AesChunkCipher` for chunk-text encryption at rest.
- **HKDF** — RFC 5869 Extract-then-Expand KDF over HMAC-SHA-256.
- **Session** — one client↔CVM RATLS round trip (ingest or query).
  Secrets live only for its duration.
- **`tenant_id`** — opaque utf-8 string identifying a customer / corpus /
  trust domain; controls key partitioning.

## 1. What we change vs the runner today

| Layer | Today | Under Option 3 |
|---|---|---|
| Embedding scheme in runner | `IdentityScheme` (pass-through) | `Caprise` with seed derived per session |
| Chunk text encryption | `AesChunkCipher::generate()` (random per process) | `AesChunkCipher` keyed from HKDF of same secrets |
| Key lifecycle | random AES key persists for process lifetime | both keys re-derived per request from `(user_x_sk, tee_user_x_sk)`; zeroized after request |
| Multi-tenancy | none (one index, one chunk key) | per-`tenant_id` index + per-tenant TEE half-secret |
| HTTP surface | `/ingest`, `/query` accept only payloads | additionally accept `tenant_id` + `user_x_sk` over the RATLS body |
| Attestation `scheme_identity` | CAPRISE constants + mask + shield | + HKDF policy version + tenant-binding rule |

The wire format and request shape change. The protocol math (GELO mask,
shield, U-Verify, CAPRISE construction) does not.

## 2. Trust model under Option 3

| Compromise | Recovers | Doesn't recover | Outcome |
|---|---|---|---|
| Untrusted storage operator alone | CAPRISE ciphertexts, access patterns | `caprise_seed_key`, `user_x_sk`, `tee_user_x_sk` | safe |
| Untrusted GPU + host alone | masked activations | clean activations, any key material | safe (GELO baseline) |
| TEE seal broken (memory dump, CVM rollback) | `tee_user_x_sk` (if persisted at the moment) | `user_x_sk` — never persisted; TLS was forward-secret | **safe — past sessions remain private** |
| Client endpoint compromise | `user_x_sk` | `tee_user_x_sk` | safe — caprise_key needs both halves |
| TEE seal **and** client compromise simultaneously | both halves | — | broken — accepted in the design |
| Active malicious CVM exfiltrating `user_x_sk` mid-session | both halves | — | broken; attestation binds **code identity**, not runtime behavior. Mitigation: pin `MEASUREMENT` in relying-party policy. |

The headline forward-security claim is the bold row: **a future TEE
break does not retroactively decrypt the index, because `user_x_sk` was
never in the TEE long enough to seal, and the TLS that carried it is
ECDHE-forward-secret.**

## 3. Key derivation

One HKDF-Extract, two HKDF-Expand calls — all sub-keys derive
deterministically from the same `(user_x_sk, tee_user_x_sk, tenant_id)`
tuple. **No hand-rolled crypto** — every primitive is a thin call into
an audited Rust library.

### 3.1 Cryptographic dependencies (all already audited libraries)

| Primitive | Crate | Org | Notes |
|---|---|---|---|
| HKDF (Extract + Expand) | `hkdf 0.12` | RustCrypto | new dependency · pure-Rust · MIT/Apache-2.0 |
| HMAC-SHA256 (HKDF backbone) | `hmac 0.12` | RustCrypto | **already in workspace deps**; used by `core/src/prf.rs` |
| SHA-256 | `sha2 0.10` | RustCrypto | **already in workspace deps** |
| AES-256-GCM (chunk text) | `aes-gcm 0.10` | RustCrypto | **already in workspace deps**; existing `AesChunkCipher` |
| Secret zeroization | `zeroize 1.x` | RustCrypto | new dependency · `Zeroizing<T>` wrapper |
| TLS 1.3 ECDHE (RATLS transport) | `rustls` via `reqwest/rustls-tls` | rustls.dev | **already in workspace deps** |
| SEV-SNP report parsing + sig verify | `sev 7` | virtee (AMD-blessed) | **already in workspace deps** |
| ECDSA-P-384 (report sigs, mock chain) | `p384 0.13` | RustCrypto | **already in workspace deps** |
| OS RNG (`tee_user_x_sk` generation) | `rand::OsRng` | rust-random | **already in workspace deps** |

The plan adds **two** new dependencies (`hkdf`, `zeroize`); both are
maintained by the RustCrypto org alongside `hmac`/`sha2`/`aes-gcm` we
already use. No hand-rolled primitives at any layer.

### 3.2 Derivation, as called

Conceptually:

```
prk = HKDF-Extract(salt = tenant_id, ikm = user_x_sk ‖ tee_user_x_sk)
caprise_seed_key = HKDF-Expand(prk, info = "gelo-rag.v1.caprise.seed", L=32)
aes_chunk_key    = HKDF-Expand(prk, info = "gelo-rag.v1.aes-chunks",   L=32)
```

In code (sketch — exact Rust shape resolved in M2):

```rust
use hkdf::Hkdf;
use sha2::Sha256;
use zeroize::Zeroizing;

pub fn derive(
    user_x_sk:     &Zeroizing<[u8; 32]>,
    tee_user_x_sk: &Zeroizing<[u8; 32]>,
    tenant_id:     &[u8],
) -> (Zeroizing<[u8; 32]>, Zeroizing<[u8; 32]>) {
    // Concatenate the two halves into the IKM. The `Zeroizing` wrapper
    // ensures the joined buffer is wiped on drop.
    let mut ikm: Zeroizing<[u8; 64]> = Zeroizing::new([0u8; 64]);
    ikm[..32].copy_from_slice(user_x_sk.as_ref());
    ikm[32..].copy_from_slice(tee_user_x_sk.as_ref());

    // RustCrypto `Hkdf<Sha256>::new(...)` IS the HKDF-Extract step.
    let hk = Hkdf::<Sha256>::new(Some(tenant_id), ikm.as_ref());

    let mut caprise_seed: Zeroizing<[u8; 32]> = Zeroizing::new([0u8; 32]);
    hk.expand(b"gelo-rag.v1.caprise.seed", caprise_seed.as_mut())
        .expect("32 ≤ 255·HashLen for SHA-256");

    let mut aes_key: Zeroizing<[u8; 32]> = Zeroizing::new([0u8; 32]);
    hk.expand(b"gelo-rag.v1.aes-chunks", aes_key.as_mut())
        .expect("32 ≤ 255·HashLen for SHA-256");

    (caprise_seed, aes_key)
}
```

Every cryptographic step in this function is a single line of library
call — no manual HMAC construction, no manual PRF chaining, no key
material on the stack outside a `Zeroizing` wrapper.

### 3.3 Design notes

- **Salt = `tenant_id`**: gives cross-tenant key separation even if a
  single `tee_user_x_sk` were ever shared. Cheap; eliminates a class of
  key-reuse mistakes.
- **Info-string versioning**: `gelo-rag.v1.…` lets us rev the scheme
  without colliding old/new ciphertexts. Bumping `v1 → v2` is a
  one-character change that yields entirely different keys.
- **No randomness at derive time**: same inputs ⇒ same keys, every
  call. That determinism is what lets the query path re-derive the same
  `caprise_seed_key` the ingest path used.
- **CAPRISE non-secret parameters** (`scale_factor`, `beta`): scheme
  constants, part of `scheme_identity`, not per-tenant secrets. Today:
  `scale=32.0, beta=0.15`. Any change requires re-attestation.
- **Single-key alternative (rejected)**: deriving one key and using it
  for both CAPRISE seed and AES is unnecessary entanglement; one
  `hk.expand(...)` call per sub-key is free.

## 4. Wire shape

### Attestation handshake (already exists, unchanged math)

```
client → cvm:   GET /attest  { nonce }
cvm    → client: SnpEvidence { report_bytes, vcek_cert_pem,
                               model_identity, scheme_identity }
client:          SnpAttestationVerifier.verify(...)
                 — pins model_id + scheme_id + measurement
```

### New per-session payload

```
POST /ingest  (over RATLS body, AEAD-protected by TLS 1.3)
{
  tenant_id:  "acme-legal",
  user_x_sk:  <base64-32B>,
  chunks:     [{ id, text }, ...]
}

POST /query
{
  tenant_id:  "acme-legal",
  user_x_sk:  <base64-32B>,
  text:       "...",
  top_k:      5
}
```

Both endpoints assume the client has already verified an attestation
report bound to this connection's TLS public key — the existing RATLS
construction. The `user_x_sk` field is the only secret on the wire.

## 5. The `tee_user_x_sk` lifecycle

Three sub-decisions, called out separately because they have different
deployment-risk profiles:

### 5.1 Generation

On first contact with a new `tenant_id`, the CVM samples
`tee_user_x_sk = OsRng(32)` and records it in a per-tenant table.
Subsequent sessions look up the existing entry.

### 5.2 Persistence

Three sub-options, ordered by simplicity:

| Variant | Where it lives | Re-launch behavior | Suitable for |
|---|---|---|---|
| **A — in-memory only** | RAM (encrypted by SEV-SNP automatically) | lost on restart; client must re-bootstrap tenant (new corpus) | T1 / T2 / prototype demos |
| **B — sealed to disk via attested KMS** | encrypted disk volume; unlock key released by KMS to attested CVMs only | survives CVM restarts at the same `MEASUREMENT` | T3 production target |
| **C — escrowed at the relying party** | client (RP) re-injects on every CVM boot via attested channel | survives but requires RP coordination | multi-tenant SaaS |

The plan ships **Variant A** in the first milestone; **Variant B** is a
separate later milestone gated on standing up a Constellation-style
attested KMS service. **C** is not currently planned.

### 5.3 Rotation

Each tenant can request rotation by:
1. POST `/rotate { tenant_id, old_user_x_sk, new_user_x_sk }`
2. CVM re-derives old keys (decrypt existing index), re-encrypts under
   new keys, replaces in storage atomically, then forgets the old
   `tee_user_x_sk` and generates a new one bound to `new_user_x_sk`.

Out of scope for the first milestone — call out in the plan but defer
implementation.

## 6. Service-level shape

`GeloRagInMemoryService` currently looks like:

```rust
pub struct GeloRagInMemoryService<E, S, V> {
    embedder: E,
    scheme: S,                         // S = Caprise / Identity / …
    verifier: V,
    chunk_cipher: AesChunkCipher,      // one fixed key for the whole process
    index: InMemoryEncryptedIndex,     // one global index
}
```

The Option-3 service is **multi-tenant by construction** and **does not
hold the per-request scheme**:

```rust
pub struct GeloRagTwoPartyService<E, V> {
    embedder: E,
    verifier: V,
    tee_secrets: Mutex<HashMap<TenantId, TeeUserXsk>>,   // §5.1
    indices:     Mutex<HashMap<TenantId, InMemoryEncryptedIndex>>,
    scheme_params: SchemeParams,         // scale, beta — public consts
    hkdf_policy:   HkdfPolicy,           // info strings + version
}
```

Per-session derivation happens inside `ingest_chunks_for` /
`query_for`:

```rust
fn ingest_chunks_for(
    &self,
    tenant_id: &TenantId,
    user_x_sk: Zeroizing<[u8; 32]>,        // dropped + zeroized at return
    chunks: Vec<DocumentChunk>,
) -> Result<()> {
    let tee_sk = self.tee_secret_for(tenant_id);
    let (caprise_seed, aes_key) =
        self.hkdf_policy.derive(&user_x_sk, &tee_sk, tenant_id);
    let mut caprise = Caprise::new(CapriseKey::from_seed(
        self.scheme_params.scale,
        self.scheme_params.beta,
        caprise_seed,
    ));
    let aes = AesChunkCipher::from_key(aes_key);
    // … existing embed + encrypt + index path, scoped to this tenant
    // both `caprise_seed` and `aes_key` drop here and zeroize.
}
```

Surface-level constraints driving this shape:

- **No long-lived `scheme: S` field.** The whole point of Option 3 is
  that the CVM has no key material at idle. A `Caprise` instance carries
  the seed — keeping one on the struct defeats the purpose.
- **`Zeroizing<[u8; 32]>` wrapper** from the `zeroize` crate for the
  client-held half and every derived sub-key. Drop-time zeroize is the
  difference between "key material in core dump" and "key material gone
  by the time anything observes the CVM".
- **Per-tenant `InMemoryEncryptedIndex`.** Cross-tenant scans become
  type-unrepresentable.
- **`verifier`** stays on the struct — it's the relying-party-side trait
  used for self-tests, not request-scoped.

The existing CAPRISE construction (`Caprise::new(CapriseKey { scale,
beta, seed_key })` in `crates/core/src/caprise.rs`) is reused
unchanged. We add **one constructor**:

```rust
impl CapriseKey {
    pub fn from_seed(scale: f32, beta: f32, seed: [u8; 32]) -> Self {
        Self { scale_factor: scale, beta, seed_key: seed }
    }
}
```

`AesChunkCipher` gets an analogous `from_key(key: [u8; 32]) -> Self`.

## 7. End-to-end flows

### 7.1 Ingest flow

```
CLIENT                                    CVM                                  STORAGE
──────                                    ───                                  ───────

(once per tenant)
  user_x_sk ← OsRng(32B)
  save to client keyring

(per ingest)
  attest:
    GET /attest?nonce=N  ───────────►
                                         build REPORT_DATA(model_id ‖
                                                           scheme_id ‖ N)
                                         issuer.issue(report_data)
                          ◄──────────  SnpEvidence{report, vcek, ids}
  SnpVerifier.verify(...) pins:
    measurement, model_id, scheme_id

  POST /ingest  (RATLS body)
   { tenant_id, user_x_sk_b64,
     chunks: [...] }       ────────────►
                                         tee_sk = tee_secrets
                                            .entry(tenant_id)
                                            .or_insert_with(OsRng(32B))
                                         prk = HKDF-Extract(
                                             salt = tenant_id,
                                             ikm  = user_x_sk ‖ tee_sk)
                                         caprise_seed =
                                             HKDF-Expand(prk, "…caprise.seed")
                                         aes_key      =
                                             HKDF-Expand(prk, "…aes-chunks")

                                         for chunk in chunks:
                                           e   = embedder.embed(chunk.text)
                                                  // GELO mask path,
                                                  // GPU offload, etc.
                                           ce  = Caprise{seed=…}
                                                    .encrypt_document(e)
                                           cct = AesGcm{key=…}
                                                    .encrypt_chunk(chunk)
                                                          ────────►  store (cct, ce)
                                                                       under tenant_id

                                         zeroize(user_x_sk,
                                                 prk,
                                                 caprise_seed,
                                                 aes_key)
                          ◄──────────  200 OK { count }
```

### 7.2 Query flow

```
CLIENT                                    CVM                                  STORAGE
──────                                    ───                                  ───────
  (attest as before — fresh nonce)

  POST /query  (RATLS body)
   { tenant_id, user_x_sk_b64,
     text, top_k }         ────────────►
                                         tee_sk = tee_secrets[tenant_id]  // EXISTING
                                         prk = HKDF-Extract(…)             // SAME as ingest
                                         caprise_seed = …                  // SAME 32B
                                         aes_key = …                       // SAME 32B

                                         q  = embedder.embed(text)
                                              // GELO mask path
                                         eq = Caprise{seed=…}
                                                 .encrypt_query(q)
                                                 // note: query coeff (1/8)
                                                 // not doc coeff (3/8)

                                         indices[tenant_id].search(eq, k')
                                                          ────────►  top-k' (cct, ce)
                                                          ◄────────
                                         for (cct, ce) in hits:
                                           e_clean = caprise.decrypt_document(ce)
                                           // OPTIONAL: rerank against
                                           // clean q for tightness
                                           chunk  = aes.decrypt_chunk(cct)
                                           result.push(RetrievalHit{
                                             id: chunk.id,
                                             score: cosine(q, e_clean),
                                             text: chunk.text })

                                         zeroize(user_x_sk, prk, …)
                          ◄──────────  200 OK { hits: [...] }
```

**Key symmetry**: at step "prk = HKDF-Extract(…)" the ingest and query
flows are byte-identical. The deterministic derivation is what lets us
encrypt at ingest with one connection and decrypt at query through a
different connection without ever persisting the key material between
them.

**Why the TEE returns plaintext hits over RATLS**: the client cannot
re-derive `caprise_seed_key` itself — it doesn't hold `tee_user_x_sk`.
So either (a) the TEE decrypts and returns plaintext over the
attested channel, or (b) the TEE returns ciphertext and a one-shot
key under a tighter wrap. (a) is simpler and aligned with the
threat model — the RATLS channel is the trust seam, and the client
is already trusted with the final result.

## 8. Attestation binding

`scheme_identity` (today: mask seed config + shield config + OutAttnMult
policy) extends to commit to the KDF parameters and the CAPRISE
constants. Canonical encoding so two CVMs configured identically produce
byte-equal `scheme_identity`:

```
scheme_identity = sha256(
    "gelo-rag.v1\n"
  ‖ "kdf=hkdf-sha256\n"
  ‖ "kdf-info-caprise=gelo-rag.v1.caprise.seed\n"
  ‖ "kdf-info-aes=gelo-rag.v1.aes-chunks\n"
  ‖ "kdf-salt=tenant_id\n"
  ‖ "caprise-scale=32.0\n"
  ‖ "caprise-beta=0.15\n"
  ‖ "mask-config-digest=" ‖ hex(mask_cfg.digest()) ‖ "\n"
  ‖ "shield-config-digest=" ‖ hex(shield_cfg.digest()) ‖ "\n"
)
```

This commits the CVM to a specific KDF policy. A relying party verifying
the report knows: the binary will derive `caprise_seed_key` from
HKDF-SHA-256, will use these info strings, and won't substitute another
KDF or another set of CAPRISE constants without re-attestation.

The `tenant_id`s themselves are **not** committed in `scheme_identity` —
they're per-request and can grow over time without re-attesting.

## 9. Failure modes & footguns to call out in code review

1. **Logging `user_x_sk`**. A debug log of the `/ingest` payload trivially
   defeats the design. Add a `#[derive(Debug)]` blocker (manual `Debug`
   impl that prints `<redacted>`) at the request-type level.
2. **Persisting `user_x_sk`**. Any code path that writes the secret to
   disk — including a panic backtrace that captures a request struct —
   degrades to "TEE-break recovers everything". The `Zeroizing` wrapper
   does not prevent OS-level swap; CVM swap should be disabled at boot.
3. **Per-call CAPRISE non-determinism**. CAPRISE has per-encryption
   nonces (the random 16-byte `nonce` field on `EncryptedEmbedding`).
   These do **not** need to be reproducible across sessions — they're
   sampled fresh per call. The seed key is what's deterministic.
4. **Tenant isolation bug**. Forgetting to scope the index by tenant
   produces a system where the wrong client's queries return the right
   client's docs — silently. Use a typed `TenantId` wrapper, not a
   bare `String`.
5. **HKDF version downgrade**. Bumping the `info` string is the safe
   migration; a downgrade lets an attacker who knows `v1` keys recover
   `v2` plaintext if both are accepted simultaneously. Pin the policy
   in the attestation report (§8) and reject mismatches at handshake.
6. **Active malicious CVM**. Attestation guarantees code identity, not
   runtime honesty. A relying party must pin `MEASUREMENT` to a build
   they trust; mock-mode runners must never share `MEASUREMENT` values
   with production-mode runners.

## 11. What this plan does **not** change

- The GELO mask / shield / U-Verify / OutAttnMult math. Untouched.
- The `Embedder` and `TrustedExecutor` traits. Untouched — both
  CAPRISE encryption and key derivation happen *after* `embed()`
  returns.
- The `EmbeddingEncryptionScheme` trait — CAPRISE still implements it,
  we just construct it per-request instead of holding one instance for
  the process lifetime.
- The `WgpuVulkanEngine` and the GPU offload path. The GPU sees masked
  activations; it never sees the CAPRISE key, the embedding, or the
  ciphertext.
- `RemoteRagService` (the Recipe-C parallel stack). That's a different
  storage model (plaintext doc embeddings + Paillier) and stays
  separate.

## 12. Open questions to resolve before M1

- **Tenant id provisioning**: is `tenant_id` an opaque client-chosen
  string, or does it require server-side registration? Current
  preference: opaque + server-side rate-limited (first contact creates
  the tenant entry; subsequent contacts re-use). Need a sentence in
  the auth model.
- **What happens when the CVM forgets a tenant**: in Variant A, a
  CVM restart loses every `tee_user_x_sk`. From the client's
  perspective the index becomes undecryptable. Acceptable for
  prototype-shape demos; the production answer is Variant B. Either
  way the failure should be **loud** (HTTP 410 Gone, distinct status
  code) so the client doesn't quietly re-encrypt under a fresh
  `tee_user_x_sk` and end up with a half-old half-new index.
- **`user_x_sk` derivation on the client side**: out of scope for this
  document — the client may itself derive it from a passphrase via
  Argon2id, store it in OS keychain, or hold it in a hardware token.
  The CVM is agnostic.
