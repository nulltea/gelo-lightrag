§01

Role in the project

where this fits · what it closes · what currently runs

Generation is the final stage that closes the private-RAG loop: embed the query, retrieve ciphertext candidates, rerank them inside the CVM, and generate the answer inside the same trust boundary rather than handing decrypted top-k chunks to a client-side LLM. Model weights are public; the protected asset is the user's prompt, the retrieved chunk text, the per-token logits, and the sampled tokens — all of which must stay confidential against a co-located untrusted GPU and an untrusted CVM operator.

runs today

Greedy decoding on Qwen/Qwen3-1.7B end-to-end under the GELO protocol: prefill + 8-token decode with bit-identical token output across PlaintextExecutor, InProcessTrustedExecutor (per-forward Haar mask + shield(8, 4.0)), and U-Verify k = 2. GELO mask is +33 % TPOT on AMD RADV GFX1151 iGPU; argmax stability holds. Qwen3-4B rides the same code path as the stretch target.

open

(a) U-Verify unusable at 1.7B-scale — k = 2 Freivalds probes against ~13 GB collapse decode to 0.04 tok/s. Needs sublinear-in-weight-size integrity primitive. (b) Decode-phase KV-cache bandwidth dominates past ~4k tokens — SCX-style per-position encoding (Yuan et al., SIGCOMM 2025) candidate; security analysis is the gating spike. Both items in §10.

§02

Definitions & glossary

terminology introduced by this stage

Only the terms specific to generation. Protocol terms (mask, shield, U-Verify, OutAttnMult, permuted attention) are defined in embedding §05 and gelo.md.

Term Short Meaning
Prefill — Single forward pass over the full prompt that populates the KV cache. Compute-intensive, attention-dominated past ~2k tokens; the "long" workload of the two phases.
Decode — Per-token autoregressive step. The new token's query attends to all cached K, V; one new K/V pair is appended. Per-step compute is small, dispatch-bound; the "narrow" workload.
Time-to-first-token TTFT Latency from request arrival to first output token. Dominated by prefill cost.
Time-per-output-token TPOT Steady-state inter-token latency once generation starts. Dominated by decode cost.
KV cache — Materialised (K, V) for every previous position, per layer, per head. Lives in encrypted CVM DRAM; sized 2 · n_layers · n_kv_heads · n_cache · d_head. ~58 MB for Qwen3-1.7B at n_cache=4k.
QK-norm — Per-head RMSNorm on Q and K before RoPE. Qwen3's only architectural delta vs Qwen2. Stays in-TEE on plaintext; absent on Qwen2/LLaMA/Mistral so the step is a no-op there.
Grouped-query attention GQA Q has more heads than K/V; multiple Q heads share one K/V head. Qwen3-1.7B uses 2:1 (16 Q, 8 KV).
Address-bus leak — Attack class where an observer on the memory or PCIe bus learns content from which addresses are accessed, not what is read at them. Defeats mask-based confidentiality on table lookups — the index, not the value, is the secret.
§03

Threat model

what each component sees · what it cannot · what's new vs embedding/rerank

Same trust posture as embedding and reranking: hardware-attested TEE on CPU, information-theoretically blind GPU, everything else hostile. The openweight assumption holds — weights pin to a public HuggingFace revision (Qwen/Qwen3-1.7B today); SHA-256 of the loaded bytes rides as model_identity through the attestation report. Generation extends the model with two new exit channels (KV cache, per-token streaming); each is closed structurally rather than statistically.

Five exit-channel leaks specific to generation

Each closed by construction.

KV-cache content leak
KV cache lives entirely in encrypted CVM DRAM, indexed by (layer_idx, head_idx, position). GPU never touches KV bytes — attention's Q · Kᵀ and P · V run in-TEE on CPU under AOCL-BLIS; K and V never persist GPU-side. Decode-phase π handling — fresh π each step vs session-persistent π — is the decision point that gates SCX-style encoded-KV adoption; see §10.
Per-token TLS timing
Streaming output flushes one ciphertext frame per sampled token over RATLS. An on-path observer sees per-token timing; combined with public knowledge of GELO+Qwen3 forward-pass cost, this is mild side information about token entropy. Mitigations at deployment time: emit at fixed cadence padded by sleep · batch tokens into fixed-size frames · disable streaming entirely and return at EOS. Protocol surface is unchanged by the choice.
Generation-length leak
Total token count is observable from TLS frame count and connection duration. Treated as out-of-scope at the protocol layer — same posture as the reranker's k_max being a deployment property, not a request property. Fixed-length deployments (always emit n_max, padded with EOS-after-stop) are available as a wrapper.
Sampler-state leak
Sampling (greedy, top-k, top-p, temperature) runs entirely in-TEE on plaintext logits. RNG is a ChaCha20 seeded from the per-session SessionKey; GPU sees neither logits nor sample. Logits zeroized after sampling; only the sampled token_id survives to the next step.
Tied-LM-head dot-product leak
Output projection is H_final · token_embeddingᵀ over 262 144 vocab rows. Mask GEMM at this shape is unchanged from the reranker's tied-LM-head pattern — masked offload of the same form as every other linear. The vocab-row gather that follows sampling stays in-TEE — token-id-keyed lookups never leave the trust boundary.
validating result

TEE.Fail (DDR5 bus-interposer attack, Oct 2025) defeats SGX, TDX, and SEV-SNP key sealing in the offline case and uses extracted CPU-TEE keys to compromise H100 confidential GPU. The architectural choice that survives it is the protocol's per-batch fresh Haar mask: mask material exists only for the duration of one forward pass; an offline TEE compromise reveals nothing about any past or future forward until a fresh mask is sampled. Today's Qwen3 measurement validates the property end-to-end.

§04

Supported workloads & architectures

two phases · Qwen3 family

Workloads — prefill & decode

Generation has two phases with structurally different compute shapes and cost profiles. The protocol covers both, with different attention paths and different cost trade-offs. The runner exposes them as a single generate(prompt, max_tokens) entry point that does prefill once and loops decode.

Phase Shape Dominant cost Attention path Mask cadence
Prefill n_q = n_prompt attention O(n²) at long context · linear projections at short context length-based auto-switch: in-TEE for short n · fused permuted attention on GPU past threshold · OutAttnMult is the 3-dispatch fallback one Haar A for the whole prefill
Decode n_q = 1, n_kv growing dispatch overhead · KV-cache bandwidth · linear projections in-TEE always. Per-step attention is microseconds at n_q = 1; auto-switch doesn't engage. KV-cache bandwidth is what SCX in §10 optimises. one fresh Haar A per decode step
why prefill and decode want different paths

Prefill at n = 4k materialises a (heads, n, n) score tensor of ~1 GB per layer that the 3-dispatch permuted-attention path forces through device memory three times — bandwidth-bound to ~3.2 GB/layer × 28 layers ≈ 90 GB on the iGPU (~1.8 s pure traffic per prefill). The FlashAttention-style fused permuted kernel drops that to ~130 MB/layer. Decode at n_q = 1 never materialises a square score tensor — per-step attention compute is microseconds and stays in-TEE; the bottleneck is KV-cache bandwidth read at every decode step, addressed by SCX-class encoded-KV (§10).

Sampling. Greedy and top-p / top-k / temperature sampling all run in-TEE on plaintext logits. The seed is derived from the per-session SessionKey; deterministic replay (same prompt, same session) is possible at temperature = 0 for debugging and reproducibility benches without breaking confidentiality of the live request. Today ships greedy only — top-p / top-k / temperature wire in alongside the production HTTP route.

Model architectures

The shipping surface is the Qwen3 family — vanilla Qwen3ForCausalLM with GQA, full causal attention, SwiGLU FFN, tied input/output embeddings, and one Qwen3-specific addition: per-head QK-norm before RoPE. The same DecoderWeights loader and forward path that services Qwen3-Embedding under the embedder/reranker protocol handles generation. The SHA-256 of the loaded safetensors + tokenizer + the (q_norm, k_norm) presence flag rides as model_identity in every attestation report — a tokenizer drift or a re-numbered <eos> trips attestation.

out of scope

Strictly harder than the current Qwen3 path.

  • · MoE generation — Qwen3-MoE, DeepSeek-V2: routing histogram is an orthogonal exit channel; needs CryptoMoE-style balanced dispatch as a separate ~3-6 month workstream.

Qwen3-4B production state · 2026-05-26

At the production shape the prior 3-stage DCT-IV path paid six full-buffer DDR5 traverses per apply / unapply. The tile-fused cascade loads a 16-column tile once, runs all three DCT stages and all three diagonal multiplications inside L2, then writes the tile out.

why decode tok/s isn't on the prefill curve

The iGPU ceiling is the in-TEE attention bucket at decode. At B = 8 · n = 2 048 it owns 53.9 % of decode wall (rising to 63 % at single-stream, 66 % at n = 3 500) — a memory-bound per-step GQA kernel reading the cached prefix. Lifting it to iGPU compute regressed 16× at the production shape. The 40+ tok/s decode target lives on dGPU substrate: HBM ~3 TB/s kernel-side reads (~75× headroom over shared DDR5) plus persistent K/V on device. Compressing decode on iGPU isn't gated on engineering — it's gated on the memory hierarchy.

§08

Optimizations

mask families · trade-offs · what's landed

The §07 short-prompt run is cheap on the Haar QR. Production prefill at n ≥ 2k flips that picture: the QR alone takes seconds. The current executor retires Haar in favour of HD₃ (A = D₃·H·D₂·H·D₁·H) — an exactly-orthogonal cascade of three sign-flips around three in-place Walsh-Hadamard transforms, no QR, mask never materialised, O(s · log s) per column. The Walsh-Hadamard matrix exists only at power-of-two side length, so the executor leans on the shield rows it was going to add anyway: pick k directly such that n + k lands on a power of two. One mask family at every shape.

Shield-padding under batching

At batched decode each step has n = B and k = next_pow2(B + 8) − B lands the stacked axis on a power of two at every batch size. k stays ≥ 8 (paper floor); worst-case k = 2·k_base − 1 = 15.

Cost vs Haar

haar
  • sample: O(s³) Householder QR per forward
  • apply/unapply: O(s²·d) dense GEMM
  • operand to GPU: s = n + 8
  • mask storage: O(s²)
hd₃ (current)
  • sample: O(s) sign-bit draws
  • apply/unapply: O(s·d·log s) FWHT
  • operand to GPU: s = next_pow2(n + 8) via shield-as-padding
  • mask storage: O(s) (three sign vectors)

Mask-family TTFT comparison — Qwen3-4B, n = 2 048

Strategy TTFT (ms) vs Haar Mask families in tree
Haar · dense QR 25 873 (base) 1 (Haar)
HD₃ · zero-pad to next pow2 32 012 +24 % 1 (HD₃)
DCT-IV · Bluestein at exact s 22 794 −12 % 2 (HD₃ + DCT-IV)

At pow2-aligned shapes (n = 2 040) HD₃ wins outright vs Haar at 16.29 s vs ~25.6 s (−36 % TTFT, headline). The +24 % regression above is the cost the design accepts only at non-pow2 single-stream long prefill — a regime the batched substrate structurally avoids.

how we got here

Haar shipped first (paper-parity); HD₃ replaced Haar at pow2 shapes for the −36 % TTFT win; HD₃ regressed +24 % at non-pow2 because the FWHT requires pow2 and zero-padding doubles the GPU operand; DCT-IV added as a non-pow2 branch (−12 % via Bluestein) which closed the regression but introduced a second mask family and a second kernel path; HD₃ + shield-to-pow2 retires DCT-IV by using the GELO-mandated shield rows themselves as the next-pow2 padding, giving back one mask family + one kernel path at the cost of restoring the +24 % non-pow2 regression. Worth it because the batched substrate moves the production regime away from non-pow2 single-stream prefill entirely.

Trade-offs

  1. Mix / unmix kernels are less parallel than Haar's GEMM. AOCL-BLIS scales the dense Haar mask 5.04× at threads = 16 vs threads = 1. The FWHT cascade chains log₂ s stages with a serial barrier between them. HD₃ still wins at long n because Haar's O(s³) sample cost dominates.
  2. Pow2 pad on the stacked-with-shield operand — addressed by shield-as-padding. At single-stream long prefill (n = 2 048) the formula picks k = 2 048; the GPU operand jumps to s = 4 096 and the mask GEMM cost doubles. At batched decode and rerank shapes the formula keeps k ∈ [8, 15] — no padding penalty.
  3. Threat-model parity vs Haar is unvalidated. HD₃'s orbit is a discrete 2^(3·s)-element set inside O(s); structurally different from the continuous Haar measure GELO's §3.2 / §4.3 BSS-hardness arguments are written against. HD₃ ships opt-in only (with_hd3_mask()); the default stays Haar until the AloePri + GELO §4.3 attack drivers show parity at the bench shapes for whichever family is the candidate default.

Protocol-side optimisations landed

aocl-blis multi-thread haar mask gemm

blas feature default-on in gelo-protocol; bench pins GELO_BLIS_THREADS = 16. Haar mask GEMM at s = 2 056, d = 2 048 measures 5.04× faster than threads = 1. HD₃/DCT-IV don't route through BLIS — Haar-only win.

§09

Attack harness

three conditions · twelve attacks · empirical HD₃ gate

Threat model: GELO §3.2 BSS-hardness game on a per-forward orthogonal mask. PCIe-side attacker observes U = A · H (and, for some attacks, the engine output U · W) for each linear offload of each forward pass. Each forward draws a fresh A from either the dense Haar measure on O(s) or the HD₃ Hadamard cascade (§08); shield rows k_shield = 8 at energy 4·mean‖h‖ appended to defeat Gram-leak attacks. The attacker does not see token ids on the wire and does not hold an obfuscated weight table. 64-prompt smoke gate run: 2026-05-20.

Conditions under test

Condition Executor configuration Role
C0 plain CapturingPlaintextExecutor wrapping PlaintextExecutor — no mask, no shield Control. Each attack must succeed against plain inference — if it doesn't, the attack is mis-calibrated.
C2 default InProcessTrustedExecutor::with_seed — paper-parity: per-forward Haar A + ShieldConfig(8, 4.0) The released GELO default. Acceptance threshold: TTRSR ≤ 10 % per recovery attack.
C3 HD₃ Same as C2 except .with_hd3_mask() — per-forward HD₃ cascade + shield(8, 4.0) The Hadamard-cascade alternative. Must match C2 within ±0.05 on every recovery attack to clear the default-flip gate.

Open protocol gaps

P0 · integrity

U-Verify cost at ≥ 1B parameters

Linear-in-weight Freivalds gives 150× decode slowdown at k = 2; k = 8 production probes scale worse. Candidate fixes: randomised-block Freivalds, batched-across-decode verification, or checkpoint-only attestation. Mandatory before integrity rides production; not a blocker for the current demonstrator.

P1 · decode bandwidth

SCX-style KV-cache encoding for decode

Decode KV-cache bandwidth at long context is the currently unmeasured cost on the demonstrator path. SCX (Yuan et al., SIGCOMM 2025) is the candidate primitive; gated on a security spike against Hidden-No-More / KV-Cloak adversaries.

gate · attack-harness

HD₃ default-flip — 512-prompt re-run

with anchor_ica / JD batched. 64-prompt smoke clears every reachable gate; the release-gate run is the final calibration before flipping InProcessTrustedExecutor::new default from Haar to HD₃.

lever · decode

HKDF-derived mask material for decode-step amortisation

Lever, not critical path. Landing condition: bench shows mask-sample > ~10 % TPOT at the larger hidden-dim shapes currently exercised.
