4. Delete the deprecated RayonCpuEngine adapter

Worth exploring ports & adapters

before two adapters · one is forbidden
after one adapter · seam still real (future dGPU)
Cost: 31 tests/benches re-wire to WgpuVulkanEngine. CI gains a GPU requirement on the parity suite — or those tests become #[ignore] with a feature flag. Aligns with feedback_no_rayon_cpu_engine.md + feedback_no_dual_paths.md.
Problem
The engine seam is documented as polyadaptive but only one adapter is legal in production — the other is a forbidden zombie.
Solution
Delete RayonCpuEngine and migrate the 31 call sites in one effort.
Wins
  • · "two adapters" claim becomes true
  • · locality: no second perf story
  • · tests can't accidentally use Rayon

Speculative in-process

today large files · narrow per-file surface

Per-file mass is high; per-file public surface is not. Looks like a deep system already — just spread across files.

if consolidated one deep DecoderLlm

But: attention.rs's five variants are an internal seam used by the wgpu engine + model families (Qwen3, Gemma4). Hiding them all may strand the perf work currently happening on attention paths.

Why speculative: apply the deletion test honestly — deleting attention.rs doesn't make complexity vanish, it makes nine hundred lines of attention logic homeless. This isn't a shallow chain; it's a big system. The fanout may simply be the system. Surface only if active edits keep crossing all four files for one logical change.
Hypothesis
If the four files always change together for one logical edit, depth at the file level is fake — promote a DecoderLlm facade.
Test before acting
Pull a month of git log on decoder/. If commits touch ≥3 of those files routinely, hypothesis holds.
Possible win
  • · generate ⊕ embed share one interface
  • · perf-work locality at one entry
  • · bench harness stops depending on internals

Top recommendation

candidate #1

Pull rerank() orchestration into the trait

Strong

Smallest blast radius, clearest gain. Six-of-seven steps duplicated verbatim between two adapters; no GPU dependency, no migration cost beyond gelo-reranker/. The padding+sealing logic is security-relevant — having it in one place reduces the chance that a future third reranker drifts. Tests at the trait's interface survive any implementation reshuffle.

~2 files of churn no CI changes no GPU dependency in-process — no adapter needed

Candidate #2 (LightKgStore.query_context) is the natural follow-up — same shape, similar reach, two-caller deduplication.
