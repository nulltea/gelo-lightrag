§02

Threat model

extends storage-CAPRISE · adds graph-specific surfaces

Same trust geometry as the CAPRISE path (storage page): the CVM is trusted, the storage server and network operator are not, the client endpoint is trusted for the duration of an RATLS session. GraphRAG introduces structural attack surfaces the flat vector path does not have — every entity, every relation, every per-node degree, every adjacency list is now a distinct side channel. Variant A neutralises those by routing every server-visible operation through one of three primitives: Ring-ORAM (over an encrypted HNSW or KV blob), XorMM volume-hiding multi-map (over adjacency & source-id lists), AES-GCM blob fetch (over chunk bodies).

Inherited weakness. CAPRISE remains distance-preserving inside the Compass block payload — defence in depth, not a substitute. DP-Forward composition is the next layer if that residual matters.

§03

Components

Variant A · 5 new + 3 changed crates

Each card pairs [method ↦ source ↦ what it secures ↦ threat ↦ overhead]. New crates marked with PLAN; changed crates with EXTEND.

PLAN · ORAM

ring-oramcrates/ring-oram

Ring-ORAM client + server protocol. Each bucket holds Z + S blocks (Z real + S dummy). Constant online bandwidth via the XOR trick; bulk cost is in the lazy-evicted path. Semi-honest mode for the default deployment; Merkle-tree mode for malicious storage server.

method
read_path + read_batch + evict (deferred) + flush · sorted-by-path-ID stash · client position map · AES-GCM bucket encryption keyed by HKDF child
source
Ren et al. Ring ORAM (USENIX Security 2015) — port from Clive2312/compass reference
secures
access pattern over the encrypted blob store · integrity (Merkle mode)
threat
semi-honest or malicious storage server · network observer
cost
O(log N) online · localhost ReadPath + EvictPath < 50 ms target at N = 10⁵
PLAN · ANN

compass-indexcrates/compass-index

HNSW-over-Ring-ORAM with the three Compass optimisations: Directional Neighbor Filtering (quantized hints prune neighbors before ORAM read), Speculative Neighbor Prefetch (batched ef_spec candidates per round), Graph-Traversal-Tailored ORAM (embedding + neighbor list co-located in one block · multi-hop lazy eviction · treetop caching).

method
per-block payload (node_id, [f32; D], [u32; M], padding) CAPRISE-wrapped before AES-GCM · upper HNSW layers cached cleartext in CVM · compass_init-equivalent ingest converts a plain hnsw_rs::Hnsw into the encrypted tree
source
Zhu, Patel, Zaharia, Popa — Compass, OSDI 2025 (artifact Clive2312/compass, USENIX AE: Available + Functional + Reproduced)
secures
query content · access pattern over the index · result correctness under malicious server (Merkle)
threat
storage operator with full DB access · network observer reading any RPC trace
cost
≈ 1 s LAN / query at LAION 100K · 1.1 s SIFT1M · 6.0 s TripClick 1.5M · 8.9 s MS-MARCO 8.8M (paper Tab. 3)
PLAN · EMM

xormm-emmcrates/xormm-emm

Non-lossy volume-hiding encrypted multi-map. Built statically from a full key→list map; get_batch returns each value list with response size independent of the queried key. Used for the LightRAG adjacency lists and per-entity / per-edge source_id lists.

method
cuckoo + XOR-filter hashing · per-key PRF-derived bucket location · bucket-padded volume · 1.5–2× storage over the raw multi-map · static build, fresh-tier buffer for incremental ingest
source
Patel, Persiano, Yeo — XorMM: practical volume-hiding EMM, CCS 2022
secures
(1) which key was queried, (2) the size of the returned value list — i.e., per-node degree
threat
storage operator inferring entity identity from adjacency-list length · per-session linkability via repeated lookups
cost
< 1 ms per get at 10⁴ keys (target) · rebuild on every ≥ 1024 inserts or 5 % growth

What the storage server sees, end-to-end. A stream of Ring-ORAM read_path + write_path RPCs against three trees plus two KV trees, plus XorMM bucket reads against two static EMMs, plus AES-GCM blob fetches. Path IDs are uniform-random by construction. Bucket sizes are uniform per-tree. XorMM response sizes are uniform per-EMM. Request rate is constant per RATLS session. Across two distinct queries by the same tenant, the server's view is computationally indistinguishable.

Note: ingest path. Same trust geometry, opposite direction — client streams documents + extracted KG over RATLS; TEE embeds, builds a plain hnsw_rs::Hnsw in encrypted CVM RAM, runs CompassIndex::from_plaintext_hnsw, pushes the encrypted ORAM tree to storage, builds and pushes the XorMM EMMs, writes the AES-GCM chunk blobs, then zeroizes the plaintext working set. Heavier than query — expect minutes per 100K docs — but a batch operation, not on any user-perceived path.
