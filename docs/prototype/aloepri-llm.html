§01

Role in the project

Qwen3 1.7B + stock llama-server · where this fits

Generation is the post-retrieval LLM step in private RAG. After the reranker emits its k_max-item AES-GCM bundle, the client decrypts the top-k_final chunks and assembles a prompt. AloePri is the route that lets that prompt be processed by an LLM running on commodity hardware without a TEE.

What this complements. Embedding, storage, and reranking all run inside a SEV-SNP CVM with a GELO-masked GPU offload (embedding, storage, reranking). Generation has a different cost shape: the model is 5–50× larger than the embedder, and decode is autoregressive, so per-token mask resampling and per-offload PCIe round-trips dominate wall-clock. AloePri sidesteps the cost curve by moving the entire forward pass to an untrusted server and shipping a one-time obfuscated artifact instead.

Why Qwen3 1.7B as the demonstrator model. Dense transformer with GQA(16, 8), 28 layers, hidden 2048, head_dim 128, RoPE base 1M, vocab 151 936. Open weights, small enough to iterate quickly, large enough to give meaningful accuracy numbers on MMLU / PIQA / HumanEval. It is also the same backbone used as the GELO-LLM demonstrator on the GELO route, so accuracy deltas between the two private-inference routes (GELO vs AloePri) are directly comparable.

Why llama.cpp as the serving stack. Pure C++ binary; no Python serving stack; CPU and Vulkan-iGPU paths both supported; OpenAI-compatible HTTP API plus a native /completion endpoint that accepts integer token-ID arrays directly. llama-server reads qwen3.embedding_length from the GGUF metadata, so AloePri's expanded internal dim d + 2h is transparent. No fork required for the obfuscation steps currently deployed (token-level permutation Π, embedding/head Gaussian noise, and the inter-head attention shuffle from paper §5.2.3).

What's traded. AloePri does not give the per-batch information-theoretic argument GELO provides. Its security is empirical: a static obfuscation, sized so published attacks (VMA, IA, ISA, IMA, NN, TFMA, SDA) recover under 15% of tokens at the paper's recommended hyperparameters. The deliberate trade is: accept a weaker but still-meaningful guarantee in exchange for compatibility with unmodified llama-server, fp32-only weight storage (8.6 GB), and zero protocol-aware code on the server side.

What's TODO. The downstream integration story — which retrieval shape, which prompt template, which token budget, which deployment topology the generation step sits in — is left as a stub. The protocol surface this page documents is the same regardless of how the surrounding RAG pipeline calls into it.

§02

Definitions & glossary

terms specific to this page
TermMeaning
AloePri Lin et al., Towards Privacy-Preserving LLM Inference via Covariant Obfuscation, arXiv 2603.01499 (ByteDance / Nanjing University). The protocol this page implements.
Covariant obfuscation Joint transform on input data and model weights such that the obfuscated model on obfuscated input produces obfuscated output equivalent to the plaintext pipeline. The "covariant" property is what makes the rewrite lossless: model accuracy is preserved exactly (up to noise terms αe·ε, αh·ε added for VMA defense).
τ (tau) Secret token-level permutation sampled from Sn, where n is the tokenizer's active vocabulary size (151 669 for Qwen3 1.7B; the GGUF's full 151 936-slot table has 267 padding slots that stay identity). Drawn once per deployment; held by the client; never leaves the trusted zone.
Π (Pi) Permutation matrix corresponding to τ. Applied to the embedding (Π·Wembed*) and model head (Whead*·ΠT) during offline rewrite.
Z Vocabulary mapping Z = { V[i] ↦ V[τ(i)] }. Used online to obfuscate input token IDs and de-obfuscate response token IDs.
P̂, Q̂ (key matrices) Generated by paper Algorithm 1 from secret base matrices (B, B-1, E, F, Z). Satisfy P̂ · Q̂ = Id. P̂ shape is d × (d+2h); Q̂ is (d+2h) × d. Adjacent layers' weights get right-multiplied by some P̂ and left-multiplied by the matching Q̂ so the cancellation runs through every forward pass.
Expansion size h Algorithm 1 hyperparameter. Internal hidden dim grows from d to d+2h inside each transformer block; residual-stream dim stays at d. Default h=128. Paper Figure 5 shows TPOT impact <10% out to h=512 on R1-Distill-14B.
R̂qk, Ĥqk, Ẑblock Paper Algorithm 2 intra-head transforms applied to Q/K weights: 2D rotary obfuscation, 2-block scaling, and RoPE-block-wise permutation respectively. Ẑblock is sampled by BlockPerm under a softmax decay tied to the RoPE base. Not deployed on Qwen3 — each of these transforms requires the §5.2.5 fusion of γqk to commute through the QK-norm site, and that fusion is empirically broken on Qwen3 (see §09).
Ûvo Random invertible matrix applied to V (left of Wv) and inverted on O (right of Wo). Stops V activations from being a clean intermediate the server can attack.
τkv, τgroup Inter-head permutations from Algorithm 2 §5.2.3. τkv shuffles K/V heads; τgroup shuffles Q/O heads within each GQA group. Both are part of the same offline rewrite, with no per-request cost.
αe, αh Gaussian noise coefficients applied to embedding and head weights: W̃embed = Π·(We + αe·ε)·P̂embed. Default αe=1.0, αh=0.2. Paper §7.3: at αe=0.5, VMA jumps to >30% recovery — these knobs matter.
QK-norm Per-head RMSNorm applied to Q and K vectors between the QKV projection and RoPE in Qwen3 (attn_q_norm.weight / attn_k_norm.weight, shape [head_dim=128], broadcast across heads). A training-stability fix borrowed from work like Chameleon / OLMo-2. Architecturally new in Qwen3 relative to the AloePri paper's evaluated baselines (Qwen2.5, Llama3, DeepSeek have no QK-norm), and the cause of the item-7 restriction documented in §09.
§5.2.5 fusion Paper §5.2.5 layer-normalisation transformation. Replaces RMSNorm(γ_vec) with RMSNorm(κ·I) after folding diag(γ_vec) into the adjacent linear weight. Exact in expectation under i.i.d. Gaussian-input assumption; per-input bias is bounded by the paper's eCnorm. Used for residual-stream norms (attn_norm, ffn_norm, output_norm). Does not work for Qwen3's QK-norm — see §09.
head-shuffle The subset of paper Algorithm 2 that survives Qwen3 QK-norm: per-layer τkv + τgroup applied as a feature-axis permutation on Wq / Wk / Wv rows and Wo columns. Permutes whole heads, leaving each head's internal layout (and the per-head_dim γ_qk broadcast) unchanged. Mathematically clean; no commutativity argument needed.
Offline rewrite One-shot client-side transform of plaintext weights into the obfuscated GGUF. Reads the source GGUF (Q8_0 quantised), dequantises to fp32, samples τ + key matrices + noise from seeds, applies §5.2.5 fusion of residual norms + Algorithm 1 keymat + Π + head-shuffle, writes fp32 GGUF + a .key.npz sidecar holding τ. Per-model, not per-request.
TTRSR Text Token Recovery Success Ratio. Fraction of original tokens an attacker recovers under a given attack. Lower = better privacy. Paper headline: TTRSR <15% on the broadest attack suite at recommended hyperparameters.
VMA / IA / ISA / IMA / NN / TFMA / SDA The seven attacks evaluated against AloePri. VMA / IA exploit plaintext-vs-obfuscated weight relationships; ISA / IMA / NN train inverters from observed internal states; TFMA / SDA exploit token-frequency patterns. The reference codebase ships them at src/security_qwen/.
RmDP Rényi-metric Differential Privacy. Paper §6 definition: combines α-Rényi divergence with a metric on the token-sequence space. The framework used to prove AloePri's noise budget is strictly smaller than a data-only mechanism's at equal accuracy.

The trust split here is fundamentally different from the GELO embedder / reranker. There is no hardware-rooted trust anchor in the loop: protection is structural (obfuscation hides the prompt) plus empirical (noise + key matrices defeat published recovery attacks). The trusted endpoint is just the client process that holds τ.

Internal-state inversion (paper §5.2.3) — what's defended, what isn't

The asset. AloePri's threat model protects the plaintext prompt and response, not the model weights. Model weights are public Qwen3 1.7B; the user's input is the secret. ISA / IMA / NN are the attacks against this asset that target runtime state rather than weight files.

What ISA does. The server operator (= adversary in AloePri's threat model) reads intermediate attention scores softmax(Q·KT/√dh) from GPU memory during inference. They train an inverter offline on plaintext Qwen3 1.7B over a public corpus, generating (prompt, attention-pattern) pairs. At attack time they feed the captured user-request attention patterns through the inverter and recover plaintext tokens.

Why the Π token permutation and §5.2.2 noise don't stop ISA. Algorithm 1 keymat preserves attention scores by design. The transform x → x·P̂R obfuscates the residual stream; the Q/K/V projection weights are correspondingly modified so that x·P̂R·W̃q = x·Wq — the per-token Q vector is byte-identical to plaintext. Same for K, V. Q·KT in the obfuscated run equals plaintext Q·KT byte-for-byte. The pre-trained inverter works as-is.

Paper §5.2.3's full Algorithm 2 solution. Apply per-attention-head transforms that do change Q·KT while keeping the softmax output preserved end-to-end:

  • R̂qk — block-diagonal 2D rotation per RoPE pair, same matrix on Q and K. Commutes with RoPE (both are 2D rotations on the same plane → SO(2) is abelian). (q·R̂)·(k·R̂)T = q·kT (R̂ orthogonal).
  • Ĥqk — per-RoPE-pair scaling diagonal applied to Q, its inverse to K. (q·Ĥ)·(k·Ĥ-1)T = q·kT. The diagonal must be paired (same scalar on positions i and i+half) to commute with the half-rotated RoPE layout llama.cpp uses.
  • Ẑblock — permutation of RoPE pair-positions, sampled by the dynamic_window construction that only swaps pairs whose RoPE frequencies are close. Approximately preserves attention without needing to permute the RoPE freq table on the server.
  • Ûvo — random invertible matrix on V's output and its inverse on O's input. Cancels through attn·Ṽ·W̃o. Paper specifies this; reference's static rewrite omits it.
  • τkv, τgroup — inter-head shuffle. Per-layer random permutations on which Q head pairs with which K/V head (under GQA constraints).

Net effect of the full paper Algorithm 2: Q·KT is preserved end-to-end (so the model still works) but its internal per-(head, pair-position) values are scrambled, and the head-index ordering is permuted. An ISA inverter trained on plaintext attention patterns sees a tensor whose per-element values, per-pair-position structure, and per-head-index identity all differ.

Why we ship only τkv / τgroup on Qwen3. R̂qk, Ĥqk, Ẑblock each require fusing the QK-norm γ vector backward into Wq / Wk via paper §5.2.5, so the per-element γ at the norm site becomes scalar κ and commutes with rotations / scalings / permutations. The §5.2.5 fusion is exact in expectation under i.i.d. Gaussian Q/K. Empirically on Qwen3, trained Q/K vectors are not Gaussian — they correlate with γ — and the per-input κ approximation breaks attention enough that the model degenerates to high-frequency-token loops. The intra-head transforms are therefore not deployable on Qwen3 in the current implementation (see §09 and figure 03b for the full mathematical picture).

What survives. τkv + τgroup permute whole heads. The 128 elements of one head move together, so the broadcast of γqk across heads is invariant under the shuffle — no fusion needed, no Gaussian assumption involved. Mathematically exact. Empirically: byte-identical model behaviour to the no-Algorithm-2 baseline.

What's left exposed. An ISA attacker on Qwen3 sees attention patterns whose per-position-pair Q·KT values are identical to plaintext, only the head-axis is permuted. With per-layer head shuffles drawn from S8 × S2 (8! · 2!8 ≈ 40K per-layer entropy, but tied across layers when the attacker compares hidden states across layers) the attacker still has to guess the per-layer head permutation. That's a substantially weaker barrier than the paper's full construction; whether it keeps ISA TTRSR ≤ 15 % on the obfuscated Qwen3 1.7B artifact is the load-bearing question the §08 harness will answer once the M2.7 run lands.

Not covered. Model architecture (the server can fingerprint Qwen3 from layer count and tensor shapes — only the weights and token identities are obscured). Query rate and timing. Tokenizer roundtrip identity — the client uses the native /completion endpoint with integer arrays so the wire payload doesn't depend on tokenizer detok→retok, but cross-version tokenizer drift between client and server is still a deployment hazard. Model-weight privacy — the client needs plaintext weights to perform the offline rewrite; AloePri does not protect the weights themselves from the client.

Qwen3 1.7B is the current target · Gemma 4 deferred

The covariant-obfuscation construction is architecture-generic per paper §8, but every new architectural feature needs its own composition-theorem application. The list below is what's actually validated end-to-end, what's deferred, and why.

Gemma 4 E2B / E4B deferred 35 / 42 · 1536 / 2560 · 256 — Blocker: 5 residual norm sites per block. Gemma 4 inserts post-norms (post-attention, post-FFN, per-layer-post-norm) and the PLE-output norm in addition to the standard pre-norms, giving roughly 5 norms per block × 35–42 blocks = 175–210 residual norm sites. Each is a §5.2.5 fusion point whose per-input κ bias accumulates multiplicatively through layer Lipschitz factors. The paper's accuracy bound is measured at 2 norms / block (Qwen2-class); at 5 norms / block the same eCnorm compounding plausibly pushes accuracy loss to 7–12 % (or worse) on a model that's also asked to swallow PLE + K=V untie + p-RoPE on top.

Other Gemma-4-specific deltas — PLE table vocab-axis permutation, K=V untying in global layers, p-RoPE-restricted R̂qk — are individually tractable but interact with the κ-compounding ceiling. Re-attempt after the Qwen3 attack-resistance benchmark is in hand and the per-norm-site κ tuning work (see §09) has been spiked.
Qwen3-MoE / Gemma 4 26B A4B research stream — MoE Require CryptoMoE balanced-dispatch defense composed on top of AloePri. Separate work item.
Multimodal (audio / vision encoders) out of scope — — Open question on per-modality attack classes; see round-2 §D.9.

Why Qwen3 1.7B. Small enough to iterate quickly (the offline rewrite plus a smoke test cycles in ~15 s on this hardware), large enough to give meaningful accuracy numbers on MMLU / PIQA / HumanEval, and the same backbone used by the GELO-LLM demonstrator on the GELO route — so accuracy deltas between the two routes are directly comparable. Stock llama-server handles the obfuscated GGUF without a fork.

The Qwen3-specific architectural delta. The AloePri paper §7.1 explicitly lists Qwen3 in its evaluated models, but the public reference (sheng1feng/Aloepri @ 60e8ea3) imports transformers.models.qwen2.modeling_qwen2 throughout — there is no Qwen3 path in the released code. The one architectural feature that distinguishes Qwen3 from Qwen2.5 at the attention block is QK-norm: a per-head RMSNorm on Q and K, sitting between the QKV projection and RoPE. The paper's Algorithm 2 (§5.2.3) has no construction for it. ByteDance's industrial deployment presumably solved this internally; the academic artifact didn't ship that solution. §09 documents the resulting gap and our partial workaround.

obfuscate_qwen3_ggufpython/path-2 :: rewriter

One-shot CLI that consumes a plaintext Qwen3 GGUF and emits an obfuscated GGUF plus a .key.npz file. AloePriClientpython/path-2 :: aloepri_client

The trusted-side wrapper. Holds τ and the tokenizer. Every complete(prompt, …) call: tokenize plaintext → map IDs through τ → POST as an integer array to llama-server's native /completion → receive tokens array → map through τ-1 → detokenize → return plaintext.

endpoint
POST {LLAMA_URL}/completion with {"prompt": [obf_ids], "return_tokens": true, …} — native llama.cpp API.

The wire payload is an int array, not text. No tokenizer encode↔decode roundtrip on the network path; no BPE-edge-case risk. The client's tokenizer is the only one in the loop.

τ permutes only the tokenizer's used vocab range [0, 151 669). The GGUF's full table size 151 936 includes 267 padding slots that stay identity. Client refuses to run if tokenizer vocab_size ≠ key.active_size.

per-request — what crosses the wire · per-block — what the server actually computes

Two figures. FIG. 03a traces one full request end-to-end, showing what is plaintext, what is obfuscated, and where the trust boundary sits. FIG. 03b shows the partial paper-§5.2.3 (Algorithm 2) deployment at one Qwen3 attention block — keymat + inter-head shuffle applied, intra-head transforms (R̂qk, Ĥqk, Ẑblock) not — focusing on the non-linear-norm constraint that forced the partial implementation.

FIG. 03a — One request through AloePri · plaintext stays inside the trusted zone · only an obfuscated integer-ID array crosses the wire navy = client (trusted) · gpu-tone = llama-server (untrusted) · dashed amber = obfuscated wire payload

Why the integer-array wire matters. The canonical AloePri protocol in the paper passes text on the wire (client detokenises obf_ids → string, server re-tokenises → ids). That works only if the tokenizer is round-trip-lossless on every ID list — a property that fails on byte-fallback tokens, special tokens, and pathological BPE merges. llama-server's native /completion endpoint accepts integer arrays directly and can return them with return_tokens: true, removing the roundtrip entirely. The wire payload is an int array; if an observer detokenises it with the stock Qwen3 tokenizer they get UTF-8 gibberish like "いるとupgrade Printf\tmodeocyCIÓN芬", but that's their reconstruction — the protocol itself never serialises to text on the wire.

The cost is a smaller obfuscation group at the attention layer — paper Algorithm 2 fully applied gives a much larger group of admissible (Q̃, K̃) pairs at each head, while head-shuffle alone only permutes among the 16!·8!·… cardinality of head orderings, which is recoverable in principle by an attacker who knows the GQA structure.

Why obfuscated weights resist quantisation. AloePri-keymat weights are heavy-tailed per-row (max ≈ 55, std ≈ 4.7 on layer 27 of Qwen3 1.7B). Block-scale quantisation (Q8_0 stores one fp16 scale per 32 elements) loses the small values to zero when a block also contains an outlier, breaking the covariant chain. K-quants (Q6_K, Q5_K_M) have more flexible scaling but still can't preserve the precision needed. fp32 is the required production format.

Determinism verification

temperature=0.0, seed=0, -np=1, --flash-attn on produces byte-identical replicates on this hardware: 5 prompts × 3 replicates × 2 endpoints, all 30 calls produce the same output as their replicate group. The concern that Vulkan workgroup ordering or flash-attn tile scheduling would introduce per-run drift does not manifest at this scale. The accuracy deltas reported below are therefore real signal, not sampling noise within a single run.

Where the slowdown lives. The 5× decode penalty is entirely the weight-bandwidth cost of fp32 storage. Per-request client-side work (tokenise + τ-map + HTTP + τ⁻¹ + detokenise) totals well under 5 ms — the actual numpy fancy-indexing for a 32-token τ-mapping is microseconds, dominated by L2 cache. If a future obfuscation construction survives Q8_0 (e.g. via QR-projected P̂_R with tighter per-row variance), TPOT recovers to plaintext parity — the protocol surface itself imposes negligible online overhead, matching paper §7.7's claim of "identical online efficiency to plaintext inference."

What this hardware represents. Strix Halo is the upper end of consumer integrated graphics (256-bit LPDDR5x-8000, ~256 GB/s memory bandwidth). Discrete GPUs with HBM (H100 ≈ 3000 GB/s, RTX 4090 ≈ 1000 GB/s) would close the fp32-vs-Q8_0 gap further but at a different cost envelope. AloePri targets the iGPU / commodity-EPYC envelope deliberately — paper §7.7's claim that decode is "identical to plaintext" is exact only when the obfuscated artifact is in the same quantisation class as the plaintext baseline, which currently it isn't.

Two reading principles

1 — TTRSR per attack, not TTRSR overall. Each attack probes a different observable: NN / VMA / IA target weights and shallow activations, IMA-basic and IMA paper-like target the trainable hidden-state inversion surface, ISA targets the deep-layer hidden state plus attention scores, TFMA / SDA target the wire-side token-id sequence. The acceptance gate is per-attack — a low overall average can hide a single high-leaking surface. The "How to read" column above states what each attack's number specifically asserts.

2 — Plain-side TTRSR matters as much as obfuscated TTRSR. Each attack must succeed against plain inference to be meaningful as a defence test. If NN doesn't hit ~100 % on plain we have an attack-implementation bug; if it does hit ~100 % on plain and ≤ 5 % on the obfuscated artifact, we have a real defence signal. The "not covered under our threat model on plain" rows (VMA / IA / TFMA / SDA) are honest: those attacks require obfuscation to even apply, so plain has no meaningful number — not a bug, just the threat-model boundary.

The Algorithm 2 deployment gap, in detail

Restatement. Paper §5.2.3 Algorithm 2 specifies five intra-head + inter-head transforms. We deploy only the two inter-head transforms (τkv, τgroup). The three intra-head transforms (R̂qk, Ĥqk, Ẑblock) cannot be applied on Qwen3 in the current implementation.

Root cause. R̂qk is a block-diagonal 2D rotation per RoPE pair. To commute with RoPE, R̂qk must be applied to the raw projection output before RoPE. Qwen3 has QK-norm between the projection and RoPE — a per-head RMSNorm with per-head_dim γ vectors. R̂qk does not commute with per-element γ (rotation mixes pair-components; γ scales them independently). The §5.2.5 fold trick that handles this for residual-stream norms — replace γ with scalar κ and fold the per-element factor backward into the linear weight — is exact only under i.i.d. Gaussian-Q assumption. Trained Qwen3 Q/K vectors correlate with γ (the model puts more activation in high-γ dims), so the per-input κ approximation is off by enough to corrupt attention. Empirically the model degenerates to high-frequency-token loops.

Reference-code state. The public reference (sheng1feng/Aloepri @ 60e8ea3) has no QK-norm handling — all attention paths import transformers.models.qwen2.modeling_qwen2, and Qwen2.5 has no QK-norm. Paper §7.1 explicitly lists Qwen3 in its evaluated models, so ByteDance's internal codebase presumably has a fix. The academic release does not, and re-deriving from first principles hits the Gaussian-assumption wall.

What the gap costs. Head-shuffle alone scrambles only which head produced which score — the per-(position_i, position_j) Q·KT values within each head are identical to plaintext. An ISA inverter can be trained on plaintext data; at attack time it sees attention patterns whose per-cell values match its training distribution, only the head-axis is permuted. Recovering a per-layer (τkv, τgroup) ∈ S8×S2 is brute-forceable in seconds if the attacker has even crude attention-pattern signatures per head. Whether ISA TTRSR stays ≤ 15 % on Qwen3 1.7B with this partial defense is an open question; the attack-resistance benchmark will measure it.

Future work options. If the attack-resistance benchmark shows ISA TTRSR > 15 % on the partial-Algorithm-2 artifact:

  • Per-norm-site κ calibration. Replace the Gaussian-assumption κ = √(mean(γ2)) with an empirically-measured κ = meanx∼corpus(RMS(Wq(x)·γ) / RMS(Wq(x))). Requires running plaintext Qwen3 on a calibration corpus; one κ per QK-norm site (56 sites on 1.7B). This is the most paper-faithful path.
  • Structured γ-commuting R̂qk. Restrict the intra-head rotation to commute with the actual γ structure — e.g. only rotate pairs where γ_i ≈ γ_{i+half} (which RoPE-pair partners share γ). Yields a much smaller effective rotation group but doesn't require any κ approximation.
  • Modify llama.cpp. Add a "scaled-RMSNorm" op that applies a per-element correction at runtime, undoing the fold bias. Opposite of AloePri's "no infra change" design goal but available in extremis.
  • Switch backbone. The cleanest fix: pin the demonstrator to a Qwen3 variant without QK-norm, or to Qwen2.5 1.5B as a fallback. Loses architectural alignment with the GELO-LLM demonstrator (which targets Qwen3), but the full paper Algorithm 2 becomes deployable.

Decision deferred until the attack-resistance benchmark produces an empirical TTRSR number.

Other open questions

  • IFEval re-run. Initial run hit a 180 s timeout under shared-iGPU contention. After the container-hygiene fix (only spawn what's under test), a re-run with a 300 s timeout should land cleanly. Defer to a session that's already running the mini accuracy harness end-to-end.
  • Q8_0-friendly obfuscation. The fp32 requirement makes the obfuscated artifact 4× larger than the plaintext source. Possible fixes: QR-project P̂R onto the orthonormal manifold (tighter per-row distribution, friendlier to Q8_0's per-block scaling) at the cost of a smaller obfuscation group; or use a smaller h (less internal expansion, weaker obfuscation).
  • Tokenizer cross-version drift. Client uses HuggingFace's tokenizers package; server holds whatever tokenizer was embedded in the GGUF. If these drift across Qwen3 releases, the τ-mapped IDs may not decode cleanly. Current mitigation: client refuses to run if its tokenizer's vocab_size mismatches the key file's active_size. Stronger: pin both sides to a specific Qwen3 commit.
